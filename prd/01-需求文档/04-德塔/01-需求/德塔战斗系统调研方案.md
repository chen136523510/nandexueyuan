# 德塔战斗/装备/怪物系统 + 游戏性设计调研方案

> 状态：调研方案 V1 | 作者：陈梓键 | 日期：2026-07-23
> 关联：`德塔世界观.md` V2 扩展路线、`ADR-003`（塔楼爬梯设计）、`MVP需求文档`
> 决策依据：社交驱动 + 纯PVE + 20人小圈子

---

## 0. 核心设计哲学（已确认）

在深入四个系统之前，先确立三个前提——它们决定了所有后续设计的选择：

| 前提 | 决策 | 理由 |
|------|------|------|
| 留存动机 | **社交驱动** | 20 人熟人圈，留存靠共同回忆和人情味，不靠排行榜和数值碾压 |
| 战斗模式 | **纯 PVE** | 成员之间不互相攻击。熟人 PK 容易产生矛盾，聚焦合作打怪 |
| 设计尺度 | **小圈子优先** | 不套用 MMO 的副本/公会/拍卖行，设计"小圈子才好玩"的机制 |

这些前提直接决定了下文四个系统的路径选择——凡是不利于社交、拉大成员差距、需要大量陌生人填充的方案，一律否决。

## 1. 战斗系统

### 1.1 核心问题

在 3200×700 像素的 2D 侧视角大厅里，战斗该怎么发生？当前德塔大厅是开放空间（草地+塔楼外围），二层三层已渲染但闲置。战斗系统必须解决"在哪打"和"怎么打"两个问题。

### 1.2 路径对比

| 维度 | A. 大厅即战场 | B. 塔楼楼层副本 | C. 传送门副本 |
|------|-------------|---------------|-------------|
| **描述** | 怪物直接刷新在大厅地图上，玩家走过去就触发战斗 | 塔楼二/三层作为独立战斗区，爬梯进入后切换 | 通过传送门进入独立 Phaser scene，隔离战斗 |
| **改造量** | 最小：只需往现有 WorldScene 加怪物对象 | 中等：需实现楼层切换逻辑，但物理门和爬梯已有基础 | 最大：需新 scene + area 房间 + 多人同步分流 |
| **对社交的影响** | ❌ 大厅变拥挤，非战斗玩家被干扰，聊天被打断 | ✅ 战斗区隔离，大厅保持纯粹的社交空间 | ✅ 完全隔离，但战斗成员"消失"到另一个 scene |
| **世界观契合度** | 一般：大厅是学员聚集地，不应该满地怪物 | ✅ 很好：塔越高越危险，二三层是"未开放区域"，有怪物合情合理 | 一般：传送门当前是"返回学院"，再开一个门显得混乱 |
| **技术风险** | 低：现有碰撞+交互框架可直接用 | 中：楼层切换逻辑需新写（但 ADR-003 已预留爬梯+物理门基础） | 高：新 scene 的创建/销毁/同步需要全新架构 |
| **可扩展性** | 差：大厅空间有限，怪物数量和种类受限 | ✅ 好：每层可设计不同主题，三层递进 | 最好：独立 scene 无限制，但现阶段不需要 |

### 1.3 推荐：路径 B — 塔楼楼层副本

**推荐理由**：

1. **三层塔楼是现成的**：`buildTower()` 已渲染三层结构（底层大厅/中层房间/顶层哨位），物理门和爬梯已有基础代码，但注释写"二三层/梯子/层级标签已移除——等正式楼层切换功能再做"。这是拖延最久的预留坑位，战斗系统让它物尽其用。

2. **世界观无缝衔接**：德塔世界观 V2 扩展路线明确写"中层房间 + 近塔区域"、"高层哨位 + 远域探索 + 战斗系统"。塔越高越危险，怪物盘踞在未开放楼层，学员的任务是"清理楼层、解锁区域"——叙事完整。

3. **社交空间不被打扰**：一层大厅保持纯粹的社交空间（聊天/NPC 对话/传送门），成员可以自由选择"今天想聊天"还是"今天想打怪"。

4. **改造量可控**：楼层切换的核心挑战是"爬梯后怎么切到不同区域"。现有爬梯状态机（`Player.setClimbing` + `WorldScene.updateLadderState`）可扩展为"到达梯顶→触发楼层切换→加载新区域内容"。不需要新 scene，沿用同一 WorldScene，用 camera 聚焦和怪物生成逻辑切换。

### 1.4 战斗模式设计

**实时 ARPG 风格（轻量）**：

- **碰撞即伤害**：怪物身上走碰撞体 → 对玩家造成伤害。参考星露谷矿井的简单位图碰撞，不需要复杂 hitbox 判定
- **玩家攻击**：自动攻击（走近怪物自动挥砍）+ 简易技能（按数字键释放，有 CD 冷却）
- **战斗节奏**：单次战斗 < 30 秒。不拖沓，成员可以边打边在聊天框调侃
- **死亡惩罚**：极轻——掉回一层大厅，不掉落任何东西。熟人圈死不起，惩罚太重没人愿意玩
- **多人合作**：多个玩家同时碰撞同一怪物 → 伤害叠加。不需要组队系统，即兴合作

```
战斗循环：
进入楼层 → 遇到怪物 → 碰撞战斗（<30秒）→ 击杀/被击退 → 掉落材料 → 回到大厅聊天
```

### 1.5 技术关键点

| 需要新增 | 可复用 |
|---------|--------|
| 楼层切换逻辑（爬梯到顶→加载新区域） | Arcade physics 碰撞系统 |
| 怪物对象（sprite + 碰撞体 + 简易 AI） | E 键交互框架（可扩展为多对象交互） |
| 玩家 HP 显示（Vue HUD 层） | 爬梯状态机（`setClimbing`/`updateLadderState`） |
| 伤害计算逻辑 | 动画系统（`createPlayerAnimations` 可扩展攻击动画） |
| 掉落物（地面可拾取对象） | 聊天气泡系统（怪物击杀后的全服广播） |

## 2. 装备系统

### 2.1 核心问题

装备系统最大的风险不是"不好玩"，而是**阻碍社交**。传统 RPG 装备逻辑（打怪→掉装备→变强→打更强的怪）在 20 人圈子里有两个致命问题：

- **交易池太小**：20 人装备交易市场形同虚设，装备要么溢出要么稀缺
- **数值差距扩大**：先来的人装备好、后来的人追不上，新人融入困难，熟人圈出现"实力阶层"

装备系统必须换个思路：**装备不是个人资产，而是社区成就**。

### 2.2 路径对比

| 维度 | A. 传统RPG装备 | B. 无属性皮肤 | C. 共享装备池 |
|------|-------------|------------|-------------|
| **描述** | 打怪掉落装备，有等级/品质/属性，个人持有 | 打怪掉落"外观"（像素皮肤/特效），不影响数值，纯展示 | 打怪掉落材料，材料共建装备，装备全社区共享 |
| **驱动力** | 最强：数值变强是强驱动力 | 一般：展示欲是弱驱动力 | 中强：共同目标是社交驱动力 |
| **社交影响** | ❌ 拉大差距，制造实力阶层 | ✅ 零数值压力，展示即社交 | ✅ 强社交机制，一起打材料一起建装备 |
| **20人适应性** | 差：交易池太小，数值差距不可逆 | 好：每个人都可以展示自己的风格 | 最好：共同目标制造凝聚力 |
| **开发成本** | 中：需数值平衡 | 低：纯美术驱动 | 中：需设计共享规则 |
| **可持续性** | 一般：装备刷满就没动力了 | 一般：皮肤多了边际效用递减 | ✅ 好：持续新增材料+装备目标 |

### 2.3 推荐：路径 B + C 混合 — 「社区共建装备库」

**核心逻辑**：

```
打怪 → 掉落材料（公共） → 存入大厅「共享仓库」 → 材料足够 → 全社区合成一件装备（皮肤）
→ 装备展示在塔楼大厅「武器架」 → 成员可"穿"大厅里的装备（纯外观，每天限穿一次）
```

**为什么这样设计**：

1. **打怪不是为了自己变强，而是为了共同装扮德塔**。装备是"我们的"，不是"我的"。每次看到大厅武器架上的装备，成员想起的是"这是我们上周一起打 BOSS 攒材料做出来的"
2. **零数值压力**：装备不影响战斗力，穿不穿都能打怪。避免"没装备不敢进楼层"的尴尬
3. **共享规则防止滥用**：每天限穿一次（避免一人霸占），成员可投票决定下一件装备合成什么
4. **材料透明**：共享仓库显示"当前材料: X/100"，所有人都能看到进度，制造"还差一点"的集体动力

**装备类型**（纯外观，像素风）：

| 类型 | 示例 | 合成材料需求 |
|------|------|:--:|
| 头部 | 猫耳发箍、学士帽、头盔 | 50 材料 |
| 身体 | 披风、铠甲、校服 | 80 材料 |
| 手持 | 光剑、法杖、扫帚 | 60 材料 |
| 特效 | 行走足迹、光环、粒子 | 100 材料 |

**装备与 skinId 的关系**：装备是叠加在现有 skinId（1-5 套形象）之上的外观层，不冲突。成员可以先选形象（CharacterView），再穿装备（武器架交互）。

### 2.5 技术关键点

| 需要新增 | 可复用 |
|---------|--------|
| `Material`/`Equipment`/`SharedInventory` 数据库表 | Prisma 迁移模式（参考 `add_wall_tables`） |
| 共享仓库 API（材料存取/合成/装备穿戴） | E 键交互框架（武器架是新可交互对象） |
| 装备外观叠加渲染（在 player sprite 上叠加装备图层） | skinId 机制（装备也是 string id，加载逻辑一致） |
| 大厅武器架（新的可交互对象，展示已合成装备） | 现有家具摆放模式（`decorateFloor`） |

## 3. 怪物系统

### 3.1 核心问题

20 人社区需要什么样的怪物？传统 MMO 的"无限刷小怪"模式在 20 人圈有两个问题：(1) 单人可刷 → 沦为单机，没有社交；(2) 无限刷新 → 没有"今天必须来"的紧迫感。

怪物系统需要制造的是一种**社交事件感**——不是"有怪可打"，而是"今晚有大事发生"。

### 3.2 路径对比

| 维度 | A. 随机刷新小怪 | B. 世界BOSS | C. 周常怪物轮换 |
|------|--------------|-----------|---------------|
| **描述** | 定时在大厅/楼层刷新，单人可击杀 | 每天定时刷新一只强力 BOSS，需多人合作 | 每周一个主题怪物，不同机制需不同策略 |
| **社交性** | ❌ 差：沦为单机刷怪 | ✅ 强：每天上线理由，共同回忆制造机 | ✅ 中强：每周新鲜感，策略讨论制造话题 |
| **开发成本** | 低：简单刷新+AI | 中：需定时服务+多玩家同步 | 高：每周需新怪物设计+美术 |
| **可持续性** | 差：刷久就腻 | 中：同一 BOSS 反复打会疲劳 | 好：每周换新保持新鲜感 |
| **20人适应性** | 差：20人抢怪或完全没人打 | ✅ 好：3-5人合作即可击杀，不要求全员在线 | 中：需协调策略，但 20 人够用 |

### 3.3 推荐：B + C 混合 — 「每日世界BOSS + 每周主题怪」

**每日世界 BOSS**（日常）：

- **刷新时间**：每晚 20:00-22:00（成员在线高峰时段，参考 20 人微信群活跃时间）
- **出现位置**：塔楼三层（顶层哨位），需要爬梯到二层再爬梯到三层才能到达
- **击杀条件**：需要 3-5 人同时碰撞才能叠加足够伤害（单人打不动，强制合作）
- **掉落**：BOSS 材料（存入共享仓库）+ 全服公告（"陈梓键、丘序明、马逸杰合力击败了 XXXX！"）
- **BOSS 不在线时**：三层是空的，可以探索但不能战斗。避免"错过就没有"的焦虑
- **BOSS 设计**：像素风 64×64（2格），比玩家显眼但不夸张。2-3 帧待机动画。简单移动 AI（左右巡逻，检测到玩家靠近→追击）

**每周主题怪物**（周常，周日轮换）：

- **设计理念**：每周一个不同的怪物，有不同的"弱点"，需要不同策略
- **弱点机制**：比如火属性怪物怕水（需要特定楼层的水池区域引怪）、高防怪物需要多人同时碰撞破防
- **策略讨论**：每周一在群里公布本周怪物主题，成员讨论战术。制造社交话题
- **低保设计**：即使策略不对也能靠人多硬打过去，只是效率低。避免"打不过就弃疗"

**怪物命名与设计（契合男德学院调性）**：

| 怪物 | 楼层 | 类型 | 特点 | 梗 |
|------|:--:|------|------|------|
| 卷王 | 2层 | 每日 BOSS | 高攻速，不停移动 | "卷王从不休息，你也不能" |
| Deadline 怪 | 3层 | 每日 BOSS | 靠近时倒计时，到 0 全员击退 | "死线到了，快跑！" |
| 摸鱼精 | 2层 | 周常 | 隐身，需要多人同时碰撞才显形 | "摸鱼摸到隐形了" |
| 论文怪 | 3层 | 周常 | 不断分裂小论文（小怪），需要先清小怪 | "论文越写越多" |
| 社恐兽 | 2层 | 周常 | 靠近玩家会逃跑，需要围堵 | "别过来！" |

### 3.4 怪物 Sprite 方案

- **尺寸**：32×32（小怪）/ 64×64（BOSS），复用现有 tileset 切割模式
- **帧数**：2-3 帧/方向（待机+移动），参考现有 NPC 精灵 `nandetong.png` 的制作方式
- **来源**：ComfyUI AI 生成（与 R-003 玩家精灵同一管线），或从 Kenney 怪物素材包（CC0）挑选
- **fallback**：如果没有美术资源，先用色块占位（与玩家精灵色块 fallback 一致），不影响逻辑开发

### 3.5 技术关键点

| 需要新增 | 可复用 |
|---------|--------|
| `MonsterState`（Colyseus schema，同步怪物位置/状态/HP） | `PlayerState` schema 定义模式 |
| 定时刷新服务（服务端 cron 或 setInterval） | `NetworkSystem` 的 `onStateChange` diff 机制 |
| 怪物 AI 脚本（简易移动+碰撞检测） | `Player.js` 的移动逻辑（怪物可视为"AI 控制的玩家"） |
| BOSS 击杀全服公告 | `NetworkSystem` 的 `sendNpcReply` 广播模式 |
| 多玩家伤害叠加计算 | 现有碰撞系统（`physics.add.collider`） |

## 4. 游戏性（留存设计）

### 4.1 核心问题

德塔不是 MMO，不需要日活百万。游戏性的目标是：**让 20 个熟人每天有理由打开德塔看一眼，每周有 2-3 次想留下来玩一会儿**。

对于社交驱动的熟人社区，留存靠三个东西：**仪式感**（每天都想做的事）、**共同目标**（大家一起做的事）、**人情记忆**（做过之后留下的事）。

### 4.2 三大留存支柱 + 一个可选

#### 支柱 1：每日仪式感

**目标**：制造"每天登录一下"的惯性，不重但持续。

| 机制 | 描述 | 参考 |
|------|------|------|
| 每日世界 BOSS | 20:00-22:00 刷新，错过=没参与今日共同回忆 | 类似动森的每日广播、星露谷的每日占卜 |
| 德塔签到 | 每天签到获得"德塔币"（社区贡献值，不是货币），展示在个人中心 | 大多数手游的签到系统，但这里不绑定奖励 |
| 每日一句话 | 签到时可写一句话（心情/吐槽），显示在塔楼大厅的"今日留言板" | 类似 Animal Crossing 的留言板 |

**设计要点**：仪式感不绑定硬性奖励。如果签到给装备/材料，就变成了"不做就亏"的负担。德塔币是纯荣誉值（类似 GitHub 的 contribution 格子），不兑换任何东西，只是"我来过"的证明。

#### 支柱 2：共同目标

**目标**：给成员一个"我们一起做点什么"的理由。

| 机制 | 描述 | 参考 |
|------|------|------|
| 共享装备池 | 一起打材料、合成装备、装扮大厅。装备是"我们的"，不是"我的" | 泰拉瑞亚的 NPC 入驻机制（满足条件→新 NPC 搬进塔楼） |
| 塔楼建设 | 三层塔楼从"空壳"到"精装"，每层需要打怪材料才能解锁新房间/装饰 | 星露谷的社区中心收集包（全社区一起填材料） |
| 每周挑战 | 本周主题怪物，需要不同策略，群里讨论战术 | Rec Room 的每周挑战任务 |
| 塔楼扩建投票 | 下一件装备合成什么？下个房间解锁什么？全社区投票决定 | 类似 DAO 治理的轻量版 |

**塔楼建设路线图**（每一阶段需要全社区累积材料达标）：

```
当前状态：空壳三层（只有墙和地板，家具少量）
↓ 第1阶段：50 材料
二层解锁：新增「训练室」（木人桩，可练习战斗）
↓ 第2阶段：100 材料
二层解锁：新增「休息室」（沙发/茶几，可坐下聊天，文字泡特效）
↓ 第3阶段：200 材料
三层解锁：BOSS 房间正式开放（之前是"未开放区域"）
↓ 第4阶段：300 材料
大厅新增：武器架（展示已合成装备）
↓ 第5阶段：500 材料
三层新增：瞭望台（全新视野，可看到塔外远处的风景彩蛋）
```

**设计要点**：建设进度是透明可见的（大厅显示"当前材料: 87/100，还差 13 个"），制造"就差一点"的集体动力。材料不够时，自然有人喊"今晚打 BOSS 啊差材料"。

#### 支柱 3：人情记忆

**目标**：把"一起做过的事"变成"看得见的回忆"。

| 机制 | 描述 | 参考 |
|------|------|------|
| 战绩墙 | 塔楼一层新增可交互的「战绩墙」，记录每次 BOSS 击杀：日期/参与者/怪物名 | Habbo Hotel 的家具收藏（每个家具都有故事） |
| 战利品陈列 | 每次击杀 BOSS 掉落一个独特的"战利品"（装饰物），摆在大厅指定位置。累积的战利品越来越多，大厅越来越有"家"的感觉 | Rec Room 的共同回忆、魔兽世界的公会战利品展示 |
| 男德通社死播报 | BOSS 击杀后，男德通 NPC 全服广播调侃："张三今天被怪物打飞了三次，但最后还是赢了！"或者"李四一个人扛了 BOSS 半分钟，真正的猛男！" | 利用现有 NPC 广播系统，增强 AI prompt 注入击杀事件 |
| 年度回顾 | 每年年末，男德通生成一份"德塔年度报告"：谁参与 BOSS 击杀最多、最活跃时段、最受欢迎装备等 | Spotify Wrapped 的轻量版 |

**战绩墙设计**（新可交互对象）：

```
┌──────────────────────────────────┐
│  🏆 德塔战绩墙                    │
│                                  │
│  2026-07-25  「卷王」被击败        │
│  参与者：陈梓键、丘序明、马逸杰     │
│                                  │
│  2026-07-24  「Deadline怪」被击败  │
│  参与者：陈梓键、丘序明            │
│                                  │
│  ...（倒序排列，最近5条）          │
│  按 E 翻页查看更早记录             │
└──────────────────────────────────┘
```

**设计要点**：人情记忆的核心是**不可删除**。战绩记录是永久的，不因为"数据库满了"而清理。这些记录是社区的"共同历史"，每一条都是"那天我们一起做过的事"。

#### 支柱 4（可选）：轻量成长

**目标**：给成员一个"我在德塔有成长"的感觉，但不制造数值差距。

| 机制 | 描述 |
|------|------|
| 战斗等级 1-5 | 仅影响可进入的塔楼层数（1级→一层，3级→三层），无数值加成 |
| 升级方式 | 参与 BOSS 击杀即可获得经验（参与即有，不按伤害排名） |
| 等级展示 | 在个人中心和玩家头顶标签显示等级徽章（小星星/小盾牌） |
| 设计哲学 | 等级是"我去过哪里的证明"，不是"我比谁强的证明" |

**为什么是可选的**：如果等级系统让成员产生"我等级低不好意思进楼层"的焦虑，就违背了社交驱动的初衷。可以在第一版不做，等成员反馈后再决定是否需要。

### 4.3 留存设计总结

```
每天登录的原因：
  ├─ 20:00 世界BOSS（错过=没参与今日共同回忆）
  ├─ 签到+留言（看今天谁写了什么）
  └─ 看一眼共享仓库材料进度（"还差13个，今晚打吗？"）

每周登录的原因：
  ├─ 周日新主题怪物（群里讨论战术）
  └─ 塔楼建设进度（"这周能解锁训练室吗？"）

长期留存的原因：
  ├─ 战绩墙越来越长（共同回忆积累）
  ├─ 大厅越来越丰富（战利品陈列+装备展示）
  └─ 年度报告（"我们一起在德塔度过了X天"）
```

## 5. 技术可行性总览

| 系统 | 难度 | 复用什么 | 需新增什么 |
|------|:--:|---------|-----------|
| 战斗（塔楼层副本） | 中 | Arcade physics、E键交互、爬梯状态机、动画系统 | 楼层切换逻辑、怪物对象、HP 显示、伤害计算 |
| 装备（共享材料池） | 中 | Prisma 迁移模式、E键交互、skinId 机制 | Material/Equipment/SharedInventory 表、武器架交互对象、装备外观叠加 |
| 怪物（世界BOSS） | 中高 | NetworkSystem 同步、onStateChange diff、广播系统 | MonsterState schema、定时刷新服务、怪物 AI |
| 游戏性（留存机制） | 低中 | 公告系统、NPC 广播、E键交互 | 签到/战绩墙/塔楼建设状态、男德通播报 prompt 增强 |

| 表 | 主要字段 | 用途 |

| `MonsterSpawn` | id, monsterId, spawnTime, defeated, defeatedBy[] | 刷新记录 |
| `Material` | id, name, quantity（全局共享，非个人持有） | 材料库存 |
| `Equipment` | id, name, type, spriteKey, requiredMaterials | 装备定义 |
| `EquipmentLog` | id, equipmentId, userId, wornAt | 穿戴记录 |
| `BattleRecord` | id, monsterId, participants[], defeatedAt | 战绩墙 |
| `CheckIn` | id, userId, date, message | 每日签到 |
| `TowerProgress` | id, stage, currentMaterials, unlockedAt | 塔楼建设进度 |

## 6. 不做的事（明确边界）

| 不做 | 理由 |
|------|------|
| ❌ PVP | 用户已确认纯PVE，熟人圈 PK 容易产生矛盾 |
| ❌ 数值成长（装备属性/攻击力/防御力） | 拉大成员差距，新人难融入。装备只做外观 |
| ❌ 经济系统（金币/拍卖行/交易） | 20人经济体量太小，强行经济只会失衡 |
| ❌ 第一版复杂 AI（寻路/技能/阶段切换） | 先用简单移动+碰撞验证核心循环，再迭代 |
| ❌ 排行榜/伤害统计 | 制造竞争焦虑，违背社交驱动哲学 |
| ❌ 付费/内购/氪金 | 朋友圈限定社区，永不做商业化 |

## 7. 分阶段实施建议

### 阶段 1：最小可玩（MVP 战斗）

工期预估：2-3 周（单人开发）

- 二层楼层切换（爬梯到顶→显示二层区域）
- 1 只每日 BOSS（"卷王"，64×64 色块 fallback）
- 简易碰撞战斗（走近→自动攻击→BOSS 掉血→击杀）
- 掉落材料（纯数字，存入共享仓库）
- 全服击杀公告
- **验证核心假设**：20 人是否愿意每天上线打 BOSS？碰撞战斗是否好玩？

### 阶段 2：装备 + 怪物扩展

工期预估：2-3 周

- 材料合成装备（第一件装备："猫耳发箍"，50 材料）
- 大厅武器架展示 + 穿戴交互
- 第二只 BOSS（"Deadline 怪"）+ 倒计时机制
- 战绩墙（可交互对象，显示最近 5 条击杀记录）
- 德塔签到 + 每日留言板
- **验证核心假设**：共享装备池是否制造了共同目标？战绩墙是否有"回忆价值"？

### 阶段 3：周常 + 塔楼建设

工期预估：3-4 周

- 每周主题怪物（周日轮换，从 2 只开始）
- 塔楼建设路线图（阶段 1-5，全社区累积材料解锁）
- 男德通社死播报（AI prompt 注入击杀事件）
- 三层开放 + BOSS 房间
- 战利品陈列（击杀掉落装饰物摆大厅）
- **验证核心假设**：塔楼建设是否制造了长期共同目标？周常轮换是否保持了新鲜感？

### 阶段 4（可选）：轻量成长 + 年度回顾

- 战斗等级 1-5（仅影响可进楼层）
- 年度报告（德塔 Wrapped）
- 更多怪物/装备/特效
