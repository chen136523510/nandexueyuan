### R-005 黑机外包检索算力（WebSocket 长连接方案）

> 登记日期：2026-07-22 | 登记人：陈梓键 | 关联 BUG：BUG-36

#### 背景

R-002 多 Agent 协作检索 v2 上线后，全量检索（`nickname LIKE '%xxx%'` 全表扫描 51 万行）导致 2 核 2G 服务器 OOM 崩溃（BUG-36）。白机临时修复为 LIMIT 50/30，但精度严重下降。黑机（R7 9700X / 32GB DDR5）基准测试显示同样查询仅需 0.07-0.13 秒，决定将重度检索算力外包给黑机。

#### 方案选型

| 方案 | 结论 |
|------|------|
| A. frp 内网穿透 | ❌ 需自写降级逻辑，依赖 frp 进程 |
| **B. WebSocket 长连接** | **✅ 采用** — 黑机主动出站连接（不需要公网 IP），天然降级，断线自动重连 |

#### 架构

```
用户浏览器 → SSE → 云端 Express :3000
  ├─ WS Hub (:3000/search-hub) ← 黑机 WS Worker 主动连接（出站）
  ├─ orchestrator.dispatchAgent()
  │    ├─ 黑机在线 + 重度任务(person_messages/mentioned) → WS 下发 → 全量检索 → 回传
  │    └─ 离线/超时/轻量任务 → 本地 LIMIT 50（降级兜底）
  └─ 大 Agent 分析 → SSE 流式输出
```

#### 降级策略

| 场景 | 行为 |
|------|------|
| 黑机在线 + 重度任务 | 黑机全量检索（数据最完整） |
| 黑机离线/超时 | 降级本地 LIMIT 50（精度降低但不宕机） |
| 轻量任务（person_stat/topic_search） | 始终本地执行 |
| 网络中断 | Worker 5 秒自动重连 |

#### 核心文件

| 文件 | 作用 |
|------|------|
| `server/src/searchHub.js` | 云端 WS Hub（鉴权 + 心跳 + 任务分发 + 降级触发） |
| `server/src/searchWorker.js` | 黑机 WS Worker（主动连接 + 心跳 + 重连 + 全量检索） |
| `server/scripts/benchmark-query.js` | 性能基准测试（默认 vs PRAGMA 2GB cache） |
| `scripts/sync-prod-db.sh` | scp 云端 prod.db 到黑机（数据同步） |

#### 配置项

| 变量 | 位置 | 说明 |
|------|------|------|
| `BLACK_WORKER_TOKEN` | 云端 .env + 黑机 .env | WS 鉴权 token |
| `CLOUD_WS_URL` | 黑机 .env | 云端 WS Hub 地址 `ws://www.nandexueyuan.top/search-hub` |

> 登记日期：2026-07-20 | 登记人：陈梓键

#### 背景

当前首页公告栏为单条文本、纯人工维护，无法体现版本迭代历程。每次发版后用户不知道改了什么、未来要做什么。需要将版本信息同步到公告系统，让用户及时了解更新内容。

#### 需求概述

在现有首页公告栏基础上，扩展为**版本公告系统**：
- 公告栏仍展示当前最新版本摘要
- 新增「版本历史」按钮（全员可见），点击查看完整版本历史
- 公告内容由后端数据库管理，发版时由 AI 从 changelog 提取并生成公告文本写入

#### 功能点

| # | 功能 | 说明 |
|:-:|------|------|
| 1 | 版本管理 | 每次发版生成一条版本记录，含版本号、日期、更新内容、未来规划 |
| 2 | 版本号规则 | 语义化 + 日期：`v1.1.0 (2026-07-20)` |
| 3 | 公告内容分类 | 每条版本记录包含两部分：**本次更新**（新增/优化/Bug修复）+ **未来规划**（下一步计划） |
| 4 | 首页公告栏 | 展示最新版本摘要（如"v1.1.0 已发布：新增德塔NPC精灵、优化男德通检索逻辑"） |
| 5 | 版本历史按钮 | 公告栏旁加按钮，点击弹出/跳转完整版本历史列表 |
| 6 | 后端 API | 新增 `GET /api/announcement/versions` 返回版本列表；现有 `GET /api/announcement` 返回最新版本摘要 |
| 7 | 数据库 | 新增 `Version` 表（或扩展现有 Announcement 表），字段：版本号、日期、更新内容（JSON）、未来规划（JSON） |
| 8 | 发版流程 | AI 从 changelog.md 提取本次变更 -> 优化措辞为公告文本 -> 写入数据库 |

#### 公告文本风格

- 用语准确、无歧义，不引起争议
- 不需要特别官方，可以适度轻松
- 示例：

> **v1.1.0 (2026-07-20)**
>
> 📋 本次更新：
> 1. 新增了德塔 NPC 精灵及立绘
> 2. 优化了男德学院男德通模块的检索逻辑
>
> 🚀 未来规划：
> 1. 预期在德塔接入怪物系统和战斗系统
> 2. 男德学院新增导师墙

#### 现有系统衔接

- 现有 `Announcement` 表（id:1 单条记录）需迁移或扩展
- 现有 `GET/PUT /api/announcement` 接口保留兼容
- 管理后台 `MainView.vue` 的公告编辑功能需同步升级

### R-007 版本号规则规范化（x.y.z 三段式语义化版本）

> 登记日期：2026-07-22 | 登记人：陈梓键 | 优先级：P2

#### 背景

R-004 版本公告系统中版本号规则为"语义化 + 日期"，描述模糊。实际操作中多次发版未更新版本号（R-005、R-006、BUG-37 均未递增）。需明确递增规则，让每次发版都有据可依。

#### 版本号格式

`x.y.z`，如 `v1.2.3`

#### 递增规则

| 段 | 名称 | 何时递增 | 示例 |
|:--:|------|---------|------|
| **x** | 大版本号 | 引入新模块、重大架构变更 | 新增德塔模块、德塔新增战斗体系 |
| **y** | 小版本号 | 完成新需求（功能新增/优化） | 男德通检索优化、公告栏改版 |
| **z** | 修复号 | Bug 修复 | 修复进入德塔无内容、修复男德通内容丢失 |

#### 混合发版优先级

一次发版包含多种类型改动时，**只递增最高级别**：

| 发版内容 | 递增 | 示例 |
|---------|------|------|
| 大改 + 小改 + Bug | x | 新模块 + 优化 + 修 Bug → x+1，y=0，z=0 |
| 小改 + Bug | y | 新功能 + 修 Bug → y+1，z=0 |
| 仅 Bug | z | 只修 Bug → z+1 |

#### 当前版本基线

- `v1.1.0`（2026-07-20）— 已记录
- `v1.2.0`（2026-07-21）— 已记录
- 之后发版（R-005/R-006/BUG-37）未更新版本号，下次发版时统一校准

### R-008 男德墙（校园墙 / 朋友圈模块）

> 登记日期：2026-07-22 | 登记人：陈梓键 | 优先级：P1

#### 背景

男德学院是约 20 人的朋友圈限定社区。现有功能偏工具向（聊天、检索、游戏），缺少用户之间的社交互动。需要一个类似校园墙 / 朋友圈 / 贴吧的模块，让用户分享图文动态、互动交流。

#### 需求概述

新增「男德墙」模块，入口位于导航栏**男德通与男通讯录之间**。用户可发布图文动态，全站可见，支持评论和点赞。

#### 功能点

| # | 功能 | 说明 |
|:-:|------|------|
| 1 | 发布动态 | 上传 **1 张照片** + 写一段文字，发布到墙上 |
| 2 | 全站可见 | 所有登录用户都能看到墙上动态（朋友圈限定，无公开匿名） |
| 3 | 评论 | 对任意动态发表评论 |
| 4 | 点赞 | 对任意动态点赞 / 取消点赞 |
| 5 | 导航入口 | 导航栏顺序：首页 → 男德通 → **男德墙** → 男通讯录 → 德塔 |

#### 交互细节（待确认）

- 动态列表：时间倒序（最新在上）vs 热度排序？→ 暂定时间倒序
- 评论：是否支持二级回复（回复评论）？→ 暂定一级评论
- 照片：单张上限大小 / 尺寸压缩？
- 删除：发布者可删自己的动态？管理员可删任意？
- 字数：文字上限多少？→ 暂定 500 字

#### 技术方案（初步）

| 层 | 方案 |
|------|------|
| 数据库 | 新增 `Post`（动态）、`Comment`（评论）、`Like`（点赞）三张表 |
| 后端 | Express RESTful API，图片用 `multer` 上传到 `server/uploads/` |
| 前端 | 新增 `/wall` 路由 + `WallView.vue`，瀑布流或列表布局 |
| 图片存储 | 本地磁盘（参考现有 `uploads/` 方案），Nginx 静态代理 |
