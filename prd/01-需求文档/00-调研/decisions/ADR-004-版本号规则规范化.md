# ADR-004：版本号规则规范化（x.y.z 三段式语义化版本）

> 状态：Accepted | 决策人：陈梓键 | 日期：2026-07-23
> 关联：需求池 R-007、R-004 版本公告系统、ADR-001

## 背景

男德学院项目的版本号长期处于无序状态：

1. **package.json 版本号是死值**：根 `package.json` 和 `server/package.json` 的 `version` 停留在脚手架默认值 `0.0.1`，从未随发版更新，也未被任何运行时代码读取
2. **真实版本号在数据库**：线上生效的版本号是数据库 `Version` 表的记录（当前 `v2.0.0`），由 `seedVersion.js` 硬编码写入，与 package.json 完全脱节
3. **历史版本断层**：需求池称 v1.1.0、v1.2.0「已记录」，但数据库实际只有 v2.0.0 一条记录，版本历史不连续
4. **无格式校验**：`createVersion` API 只校验非空和唯一性，不校验 semver 格式，版本号全靠人工填写，存在 `2.0`、`v2`、`2.0.0` 等不一致风险
5. **多次发版未更新版本号**：R-005（黑机外包检索）、R-006（P4 角色创建）、BUG-37 等多次发版均未递增版本号
6. **无决策记录**：R-007 规则在需求池定稿，但决策依据、替代方案对比未沉淀为 ADR

## 决策

**采用 x.y.z 三段式语义化版本，带 `v` 前缀**（如 `v2.0.0`），与数据库现有格式一致。

### 版本号格式

`vx.y.z`，如 `v2.0.0`。校验正则：`^v\d+\.\d+\.\d+$`。

- 数据库 `Version.version` 字段存储带 `v` 前缀的完整版本号（如 `v2.0.0`）
- `package.json` 的 `version` 字段遵循 npm 规范**不带 `v` 前缀**（如 `2.0.0`），两者是同一版本的不同展示形式

### 递增规则

| 段 | 名称 | 何时递增 | 示例 |
|:--:|------|---------|------|
| **x** | 大版本号 | 引入新模块、重大架构变更 | 新增德塔模块、德塔新增战斗体系 |
| **y** | 小版本号 | 完成新需求（功能新增/优化） | 男德通检索优化、公告栏改版 |
| **z** | 修复号 | Bug 修复 | 修复进入德塔无内容、修复男德通内容丢失 |

### 混合发版优先级

一次发版包含多种类型改动时，**只递增最高级别**：

| 发版内容 | 递增 | 示例 |
|---------|------|------|
| 大改 + 小改 + Bug | x | 新模块 + 优化 + 修 Bug -> x+1，y=0，z=0 |
| 小改 + Bug | y | 新功能 + 修 Bug -> y+1，z=0 |
| 仅 Bug | z | 只修 Bug -> z+1 |

### 一致性约定

确立「三者一致」原则：

```
package.json version（不带 v） == 数据库最新 Version.version（带 v） == 当前线上应用版本
```

例如：package.json `"version": "2.0.0"` 对应数据库 `Version.version = "v2.0.0"`，对应线上 v2.0.0。

## 理由

1. **语义化版本是业界标准**：SemVer 被绝大多数开源项目和 npm 生态采用，团队（含未来协作者）认知成本低
2. **带 `v` 前缀**：数据库已有 v2.0.0 格式，用户在版本历史弹窗看到 `v2.0.0` 比 `2.0.0` 更直观，保持向后兼容
3. **package.json 不带 `v`**：npm 规范要求 version 字段为纯 semver，带前缀会导致 `npm version` 等工具异常
4. **混合发版只递增最高级别**：避免一次发版同时跳多个段（如 `v2.0.0` -> `v3.1.1`），版本号变化能清晰反映发版性质
5. **三者一致**：消除 package.json 死值问题，让 package.json 成为「当前应用版本」的可信来源之一

## 替代方案（已否决）

| 方案 | 描述 | 否决理由 |
|------|------|---------|
| CalVer（日历版本） | `2026.07.0` 按日期递增 | 单人项目发版不规律，日期版本无法反映改动性质（大改/小改/修复），且与 npm 生态不兼容 |
| 纯数字递增 | `v1`、`v2`、`v3` | 无法区分发版性质，用户看不到改动规模，历史版本难回溯 |
| 无前缀 semver | `2.0.0` 不带 v | 与数据库现有 `v2.0.0` 格式不兼容，需全局替换，且展示不够直观 |
| 不规范化 | 维持现状 | 版本号双轨脱节、历史断层、无校验，技术债持续累积 |

## 影响

### 发版流程变更

新增**版本号 bump 步骤**，纳入标准发版流程：

1. 确定本次发版性质（大改/小改/修复）-> 按 ADR-004 递增规则计算新版本号
2. 更新 `package.json`（根 + server）的 `version` 字段（不带 v 前缀）
3. 更新 `seedVersion.js`，新增一条版本记录（带 v 前缀）
4. 正常执行 `deploy.sh` 部署
5. 验证 `GET /api/announcement` 返回新版本号

### API 校验

`createVersion` 和 `updateVersion` API 新增版本号格式校验（正则 `^v\d+\.\d+\.\d+$`），不合规返回参数错误。

### 历史版本补录

补录 v1.1.0（2026-07-20）、v1.2.0（2026-07-21）到数据库，使版本历史连续完整。依据根 `CHANGELOG.md` 还原。

### 不做的事

- **不做前端构建期版本注入**：前端版本展示 100% 来自后端 API，注入 `__APP_VERSION__` 会引入双源不一致风险
- **不做发版 bump 自动化脚本**：单人项目手动 bump + seedVersion 已足够，自动化 ROI 低
- **不改 Version 表 schema**：现有字段（version/date/summary/updates/plans）满足需求

## 复盘条件

若未来出现以下情况，重新评估本决策：

1. 引入 CI/CD 自动化发版，需要版本号自动递增机制
2. 发版频率显著提升（月多次），手动 bump 成为瓶颈
3. 需要支持预发布版本（如 `v2.1.0-beta.1`），当前正则不支持
