# ADR-003：塔楼爬梯机制 + 物理门交互设计

> 状态：Accepted | 决策人：陈梓键 | 日期：2026-07-18
> 关联：三层塔楼改造、BUG-23（坐标系）

## 背景

德塔塔楼原为单层火柴盒（梯子+二三层因无物理碰撞已删除）。用户要求做回三层架构：
- 楼梯攀爬上楼
- 房间门可交互（物理障碍，按 E 开关）
- 室内家具装饰

核心约束：玩家跳跃极限约 90px（2.8 格），楼层高度 192px（6 格），**单靠跳跃无法上楼**。

## 决策

### 1. 上楼机制：梯子状态机（非平台跳跃）

**采用**：给 Player 加 `isClimbing` 状态，靠近梯子按 ↑ 进入爬梯（关闭重力、上下键控制 Y 速度）。

**否决方案**：
- 平台式楼梯（逐格跳）：塔楼空间被台阶占满，且 Phaser 4 单向平台支持差
- 场景切换：工作量过大，每个楼层独立 Scene
- 自动传送：沉浸感差

### 2. 门交互：物理障碍 body.enable 切换

**采用**：门初始加入 `this.ground` staticGroup（碰撞体挡路），按 E 时 `door.body.body.enable = false` 关闭碰撞 + 换贴图。

**否决方案**：
- 从 ground 移除 body：Phaser staticGroup 的 remove 可能异常
- 纯装饰门：不符合"物理障碍"需求

### 3. 梯子检测：physics.overlap

**采用**：梯子用透明 rectangle + `physics.add.existing(zone, true)` 做静态碰撞体，用 `physics.add.overlap(player, ladders)` 检测重叠，在 `updateLadderState()` 里按 ↑ 进入爬梯。

## 详细设计

### Player 爬梯状态机

```
正常状态（gravity on）
   │ 按 ↑ + 在梯子区域
   ↓
爬梯状态（gravity off, velocity Y 由上下键控制）
   │ 左右键 / 跳跃键 / 离开梯子 / 着地
   ↓
正常状态
```

- 进入爬梯：`body.setAllowGravity(false)` + `setVelocityX(0)`
- 退出爬梯：`body.setAllowGravity(true)`
- 爬梯速度 120px/s（比行走 200 慢，模拟攀爬感）

### 门状态

```
关闭（body.enable=true，挡路，door_mid 贴图）
   │ 按 E
   ↓
打开（body.enable=false，可通行，chest_open 贴图临时替代）
   │ 按 E
   ↓
关闭
```

> 临时用 `chest_open` 贴图当开门视觉，后续切专门的"打开的门"贴图。

## 理由

1. **梯子状态机是 Phaser 平台游戏的标准做法**，比平台跳跃更可控
2. **body.enable 切换比 remove 更安全**，不破坏 staticGroup 结构
3. **overlap 检测比距离判定精确**，玩家完全在梯子区域内才触发

## 影响

- `Player.js`：新增 isClimbing 状态 + setClimbing() 方法
- `InputSystem.js`：新增 S/↓ 键支持（爬梯向下）
- `WorldScene.js`：buildTower 重写为三层，新增 ladders 组、doors 数组、updateLadderState、toggleRoomDoor
- `shared/constants.js`：可能需要调 CLIMB_SPEED 常量（当前硬编码在 Player.js）

## 待验证

- [ ] 爬梯手感（速度是否合适、退出条件是否自然）
- [ ] 门开关 body.enable 切换是否真的阻止玩家穿过
- [ ] 梯子口（天花板缺一格）玩家能否爬上去不被挡
- [ ] 三层共 18 格高，相机能否跟随到顶层

## 复盘条件

如果爬梯手感差或 body.enable 有 bug，重新评估：
- 用 Arcade physics 的 `world.setBoundsCollision` 或 collider 分组
- 或改用场景切换（每层独立 Scene）
