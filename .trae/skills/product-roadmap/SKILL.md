# AI产品路线图规划（AI部门专属）

[系统级互斥锁] 严禁与其他重型技能（带流程制约的任务）并行执行。若侦测到当前上下文被要求同时运行其他技能，必须中止一切行为并要求用户降级为单线任务。

## 概述

创建和管理AI产品路线图，展示能力演进阶梯和战略方向，帮助团队和利益相关者理解AI产品发展方向。

**核心定位：** AI产品路线图不是**高铁时刻表**（精确到分钟不容偏差），而是**登山探索地图**——标明主峰方向、可行路径、补给点、危险区，允许绕行与调整。

**核心本质：** 在不确定性中建立共识，在变化中保持方向<sup>[1]</sup>。

**核心矛盾：** 我们想画一张"确定性"的路线图，却要面对一个"非确定性"的技术现实——模型能力快速升级、框架生态快速迭代、真实场景效果可能低于预期<sup>[1]</sup>。

**开始时宣布：** "我正在使用AI部门专属AI产品路线图规划技能来创建AI产品路线图。"

## 何时使用

**在以下情况使用：**
- 规划AI产品功能发布和能力演进
- 管理AI产品生命周期和数据飞轮迭代
- 与利益相关者沟通AI产品战略方向
- 制定季度或年度AI产品计划
- 优先级排序AI功能需求

**在以下情况不使用：**
- 技术实施细节讨论
- 代码审查或技术问题
- 单个AI功能的具体PRD编写

## AI产品路线图核心理念

### 三重客观约束（规划起点）

AI产品规划不能套用传统"需求—功能—排期"线性模型，必须先承认三重约束<sup>[2]</sup>：

| 约束类型 | 说明 | 规划原则 |
|----------|------|----------|
| **模型能力边界** | 生成质量、响应延迟、上下文长度随模型版本动态变化 | 以"当前基线能力"作为硬性天花板，禁止在未验证能力上设计需求 |
| **数据与合规** | 训练数据版权、敏感信息过滤、行业监管要求 | 合规设计前置，不然后果可能是产品无法上线 |
| **用户认知分层** | 从"尝鲜型"（接受试错）到"生产型"（要求零错误） | 匹配交付节奏，MVP面向早期采用者，V1保障生产可用 |

### 四步法绘制可信AI路线图<sup>[1]</sup>

1. **锚定客户价值点**，而非技术热点
   - 问：客户现在最痛的是什么？当前方案哪个环节效率最低？真是AI能解决的吗？
   - 避免：被技术牵着走，"为了AI而AI"
   - 输出：客户痛点优先级清单

2. **设计渐进式演进路径**，而非跳跃式目标
   - AI能力无法一蹴而就，把宏大目标拆解为可验证的阶段性里程碑
   - 输出：每个大目标下的技术演进阶梯图

3. **公开关键假设与取舍决策**
   - AI产品充满未知，坦诚比完美更重要
   - 明确写出：我们假设什么？我们决定不做什么？

4. **建立反馈-调整闭环机制**
   - AI路线图不是一次性交付物，是持续更新的活文档
   - 季度评审回顾假设，月度扫描技术雷达，POC小范围验证

### 三层能力成熟度模型<sup>[2]</sup>

AI产品Roadmap应按能力成熟度纵向分层，而非简单罗列功能上线时间：

| 层级 | 阶段 | 核心目标 | 关键举措 |
|------|------|----------|----------|
| **L1 基础能力层** | 0-3个月 | 稳定可用，让用户"敢用" | 标准化输入解析、确定性结果兜底（置信度<85%转人工）、基础可控性（温度/种子调节） |
| **L2 增强能力层** | 3-9个月 | 深度适配，让用户"离不开" | 领域知识注入（RAG接入私有知识库）、个性化风格学习、多轮对话状态管理 |
| **L3 智能协同层** | 9-18个月 | 主动进化，让产品"懂你未说" | 跨工具链自动调度、异常归因自修复、从用户行为反推隐性需求 |

## AI产品路线图结构

### 1. 战略锚点

**产品愿景：**
- AI产品长期价值主张
- 目标市场定位
- 解决什么核心痛点

**战略主题：**
- 当前季度/年度的战略重点
- 与公司战略的契合度
- 核心价值假设

### 2. 关键假设与取舍

**必须公开假设**（坦诚比完美重要）：

| 假设ID | 假设描述 | 验证方法 | 验证状态 |
|--------|----------|----------|----------|
| A001 | 用户愿意为私有化部署付费 | POC意向客户访谈 | 待验证/已验证/已推翻 |
| A002 | 当前模型上下文长度支持主流场景 | 跑测真实业务数据 | 待验证/已验证/已推翻 |

**必须明确取舍**（敢于说不做什么）：
- 我们决定不做：`[列出明确不做的范围]`
- 原因：`[为什么现阶段不做]`

### 3. 能力演进阶梯（按季度）

推荐结构：

| 季度 | 核心目标 | 关键举措 | 能力级别 |
|------|----------|----------|----------|
| 2025 Q2 | 降低使用门槛，提升交付效率 | • 支持Docker Compose一键部署<br>• 实现YAML生成自愈修复<br>• 上线飞书/企微报警通知 | L1 基础能力 |
| 2025 Q3 | 实现动态决策，减少人工干预 | • 支持条件分支工作流<br>• 接入LangSmith可观测能力<br>• 输出行业模板案例 | L2 增强能力 |

每个季度目标必须标注：
- `已验证` / `待探索` / `可能调整`

### 4. 数据规划（AI产品特有）

| 阶段 | 数据需求 | 标注计划 | 回流机制 | 预计完成 |
|------|----------|----------|----------|----------|
| MVM | 核心场景200条 | 内部标注 | 点赞/点踩反馈 | Q2末 |
| V1.0 | 核心场景1000条 | 供应商+内部 | 编辑行为回流 | Q3末 |

**必须包含：**
- 数据来源：自采/合作/公开/合成/用户回流？
- 隐私合规：数据脱敏方案确认
- 触发重训练条件：数据增量达到多少触发一轮微调？

### 5. 成本预算规划（AI产品特有）

| 成本项 | 月度预算上限 | 备注 |
|--------|--------------|------|
| LLM API Token | | 按预估QPS计算 |
| 向量存储 | | |
| 标注费用 | | |
| GPU算力（微调） | | 按次数估算 |

**成本控制策略：**
- 设定月度成本封顶，超阈值自动降级
- 流量高峰期自动切小模型降低成本
- 定期清理冷数据降低存储成本

### 6. 风险识别与缓解

| 风险ID | 描述 | 影响 | 可能性 | 缓解策略 | 责任人 |
|--------|------|------|--------|----------|--------|
| R001 | 模型效果不达标 | 高 | 中 | 提前跑测Golden Dataset，预留回退方案 | 算法 |
| R002 | Token成本超预算 | 中 | 高 | 设定月度封顶，自动降级切小模型 | 产品+工程 |
| R003 | 数据质量差 | 高 | 中 | 分阶段采集，边标注边验证 | 数据+产品 |
| R004 | 模型漂移 | 中 | 中 | 监控准确率，触发自动告警 | 算法+工程 |

## 优先级排序原则

### AI功能优先级定义

**P0 - 必须做（ blocking ）：**
- 核心路径基础能力不可缺少
- 合规风控强制要求
- 严重影响用户信任的问题

**P1 - 应该做（高优先级）：**
- 明确业务价值，用户强烈需求
- 构筑竞争壁垒
- 数据飞轮关键节点

**P2 - 可以做（低优先级）：**
- 改进型优化
- nice-to-have体验提升
- 探索性技术实验

### 优先级决策因素

**AI产品特有考量：**
- **数据可得性**：是否有足够高质量训练数据？
- **效果可验证**：是否能通过Golden Dataset量化评估？
- **成本可控性**：Token运行成本是否在预算内？
- **合规风险**：是否满足行业监管要求？

传统因素依然需要：业务价值、用户价值、技术可行性、战略契合度。

## AI产品路线图维护

### 闭环更新机制

| 活动 | 频率 | 目的 |
|------|------|------|
| 技术雷达扫描 | 每月 | 跟踪主流框架/模型更新，评估是否提前调整路线 |
| 数据指标复盘 | 每月 | 检查数据飞轮是否正常转动 |
| 假设验证评审 | 每季度 | 回顾关键假设是否成立，调整路线方向 |
| 客户POC验证 | 持续 | 小范围试点验证路线图可行性 |

### 发布后效果评估（AI特有）

**评估节点：**
- **MVM上线后2周**：验证核心场景准确率是否达标
- **V1上线后1个月**：评估人工介入率、Token成本、用户满意度
- **V2上线后3个月**：评估数据飞轮效果，模型迭代增益

**AI特有评估指标：**

| 指标类别 | 监控内容 |
|----------|----------|
| **模型效果** | 核心场景准确率、人工介入率、平均置信度 |
| **运行成本** | 单请求平均Token、月度总成本、成本ROI |
| **数据飞轮** | 用户反馈回流率、标注增量、模型迭代增益 |
| **模型健康** | 准确率漂移趋势、是否触发重训练阈值 |

## 路线图质量检查清单

**AI特有检查项：**

- [ ] 三重约束已识别：模型能力边界、数据合规、用户认知分层
- [ ] 关键假设已明确公开，有验证计划
- [ ] 不做什么已明确说清，避免范围蔓延
- [ ] 能力拆解为渐进式演进阶梯，不是跳跃式目标
- [ ] 数据规划完整：来源/标注/回流/重训练触发条件
- [ ] Token成本预算明确，有封顶和降级策略
- [ ] AI特有风险已识别（模型效果、数据质量、漂移）
- [ ] 每个演进阶段有可量化验证标准（Golden Dataset通过率）

**通用检查项：**

- [ ] 战略方向清晰，优先级合理一致
- [ ] 依赖关系明确，时间线考虑缓冲
- [ ] 风险都有缓解策略，资源分配合理

## 常见AI路线图误区

| 误区 | 问题 | 修复 |
|------|------|------|
| 技术堆砌 | 罗列LLM、Agent各种热点技术，不说清楚解决什么客户痛点 | 回到第一步：重新锚定客户价值点 |
| 愿景空谈 | "打造通用智能体平台"，没有可落地阶梯 | 拆解为渐进式演进路径，每个季度有具体目标 |
| 过度承诺 | 受市场压力承诺无法兑现的时间点 | 坦诚不确定性，标注"待探索"和"可能调整" |
| 忽略数据规划 | 只规划功能，不规划数据从哪来 | 补上数据规划章节，明确标注和回流 |
| 忽略成本 | 只算研发成本，不算Token运行成本 | 补上成本预算，设定封顶和降级策略 |

## 下一步

**路线图规划完成后：**
1. **对齐评审** - 与算法/工程/业务利益相关者评审确认
2. **沟通同步** - 与所有团队分享路线图，确保方向一致
3. **开始实施** - 从第一个季度目标开始实施规划
4. **持续跟踪** - 按月度/季度复盘，根据验证结果调整路线

**记住：** AI产品路线图的价值不在于完美预测未来，而在于**对齐认知**——让研发理解为什么做，让老板看到进展方向，让自己保持清醒不被技术浪潮裹挟<sup>[1]</sup>。
