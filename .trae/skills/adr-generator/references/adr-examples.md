## 示例 1: 技术选型类

### ADR-001: 统一使用 uv 管理 Python 环境

```markdown
# ADR-001: 统一使用 uv 管理 Python 环境

## 状态 (Status)
Accepted

## 背景上下文 (Context)
团队在 Python 项目开发中面临以下挑战:
1. pip 安装依赖速度慢,尤其是在 CI/CD 环境中
2. requirements.txt 无法锁定间接依赖版本,导致"在我机器上能跑"问题
3. 虚拟环境管理分散,不同项目使用不同工具(venv/pipenv/poetry)

## 决策 (Decision)
采用 uv 作为团队统一的 Python 包管理工具。
核心依据:
1. Rust 编写,依赖解析速度比 pip 快 10-100 倍
2. uv.lock 提供严格的依赖锁定,确保可复现性
3. 统一工具链,降低团队协作成本

## 替代方案 (Alternatives Considered)

### 方案 A: Poetry
**优点**: 功能全面,依赖解析准确,社区活跃
**缺点**: 启动速度慢,锁文件机制复杂
**结论**: 功能过于复杂,团队需要更轻量的工具

### 方案 B: Pipenv
**优点**: 官方推荐,锁文件机制成熟
**缺点**: 锁文件经常冲突,更新频率低
**结论**: 已有团队踩坑,不推荐继续使用

## 后果 (Consequences)
### 正面收益
- CI/CD 依赖安装时间减少 80%
- 彻底解决"在我机器上能跑"问题
- 开发者日常体验显著提升

### 负面妥协
- 团队需要学习新工具命令
- 部分 CI 镜像需要预装 uv

---
**决策人**: 向海涛 (Shawn)
**日期**: 2025-03-27
```

## 示例 2: 架构模式类

### ADR-002: 采用微服务架构拆分单体应用
```markdown
# ADR-002: 采用微服务架构拆分单体应用

## 状态 (Status)
Accepted

## 背景上下文 (Context)
现有单体应用面临以下问题:
1. 代码库庞大,构建和部署时间超过 30 分钟
2. 一个模块的故障可能导致整个系统不可用
3. 团队规模扩大,代码冲突频繁
4. 技术栈耦合,难以引入新技术

## 决策 (Decision)
将单体应用按业务领域拆分为微服务架构。
拆分边界: 用户服务 / 订单服务 / 支付服务 / 通知服务
核心依据:
1. 支持独立部署和扩展
2. 故障隔离,提升系统整体可用性
3. 团队可独立开发和交付
4. 技术栈自由,各服务可选择最适合的技术

## 替代方案 (Alternatives Considered)
### 方案 A: 保持单体架构
**优点**: 简单,无分布式系统复杂性
**缺点**: 无法解决当前面临的扩展性问题
**结论**: 不满足业务增长需求

### 方案 B: 模块化单体 (Modular Monolith)
**优点**: 保留单体部署简单性,代码层面解耦
**缺点**: 仍然无法独立部署和扩展
**结论**: 过渡方案,最终仍需微服务

## 后果 (Consequences)
### 正面收益
- 部署时间从 30 分钟降至 5 分钟/服务
- 故障隔离,单服务故障不影响全局
- 团队可并行开发,交付效率提升

### 负面妥协
- 引入分布式系统复杂性(服务发现/配置中心/链路追踪)
- 运维成本增加
- 数据一致性保证更复杂

---
**决策人**: 架构委员会
**日期**: 2025-03-15
```

## 示例 3: 数据库选型类

### ADR-003: 使用 PostgreSQL 作为主数据库
```markdown
# ADR-003: 使用 PostgreSQL 作为主数据库

## 状态 (Status)
Accepted

## 背景上下文 (Context)
新系统需要选择主数据库,需考虑:
1. 支持 ACID 事务
2. 支持复杂查询和 JOIN
3. 团队熟悉度
4. 运维成本

## 决策 (Decision)
选择 PostgreSQL 作为主数据库。
核心依据:
1. 成熟稳定,社区活跃
2. 支持 JSONB,可混合关系型和文档型存储
3. 团队有丰富的 PostgreSQL 运维经验
4. 开源免费,无厂商锁定

## 替代方案 (Alternatives Considered)
### 方案 A: MySQL
**优点**: 用户基数大,云厂商支持好
**缺点**: JSON 支持不如 PostgreSQL 灵活
**结论**: 功能上 PostgreSQL 更适合

### 方案 B: MongoDB
**优点**: 文档模型灵活,开发效率高
**缺点**: 事务支持弱,不适合金融场景
**结论**: 不满足 ACID 要求

## 后果 (Consequences)
### 正面收益
- 开发效率高,JSONB 减少表设计时间
- 性能优秀,满足预期 QPS
- 运维工具成熟

### 负面妥协
- 水平扩展不如 MongoDB 方便
- 需要专门的 DBA 支持

---
**决策人**: 后端团队
**日期**: 2025-03-10
```

## 示例 4: 前端架构类

### ADR-004: 采用 React + TypeScript 技术栈
```markdown
# ADR-004: 采用 React + TypeScript 技术栈

## 状态 (Status)
Accepted

## 背景上下文 (Context)
新前端项目需要选择技术栈,考虑因素:
1. 团队技能栈
2. 长期维护性
3. 生态系统成熟度
4. 招聘难度

## 决策 (Decision)
采用 React + TypeScript 作为前端技术栈。
核心依据:
1. 团队 80% 成员有 React 经验
2. TypeScript 提供类型安全,降低运行时错误
3. React 生态最成熟,第三方库丰富

## 替代方案 (Alternatives Considered)
### 方案 A: Vue 3
**优点**: 学习曲线平缓,模板语法直观
**缺点**: 团队经验少,生态不如 React
**结论**: 不适合当前团队

### 方案 B: Svelte
**优点**: 编译时框架,运行时体积小
**缺点**: 生态不成熟,招聘困难
**结论**: 技术风险过高

## 后果 (Consequences)
### 正面收益
- 开发效率高,团队无需额外学习
- TypeScript 类型检查减少 30% 以上低级错误
- 招聘容易

### 负面妥协
- React 本身概念较多,新成员上手需要时间
- 需要维护复杂的构建配置

---
**决策人**: 前端团队
**日期**: 2025-03-05
```
