# AI产品需求验证（AI部门专属）

[系统级互斥锁] 严禁与其他重型技能（带流程制约的任务）并行执行。若侦测到当前上下文被要求同时运行其他技能，必须中止一切行为并要求用户降级为单线任务。

## 概述

验证AI产品需求的完整性、可行性和一致性，确保开发团队能够理解和实施需求。

**核心定位：** AI产品需求验证不是"找文字毛病"，而是**工程闸门**——在开发开始前，确认四大确定性边界是否真的锁定，避免把工程不可行的伪需求带入开发阶段。

**核心原则：** 四大边界没有全部确认 → 需求不予准入，不能进入下一阶段。

**行业最佳实践依据：**
- 数据就是你的PRD，具体示例比模糊描述更重要 <sup>[1]</sup>
- 验收靠全量化测试集说话，而非主观感觉 <sup>[2]</sup>
- 权责前置，清晰切分缺陷归因，避免背锅 <sup>[2]</sup>
- AI时代评审核心是"人与智能系统的协作规则"，而非静态界面 <sup>[3]</sup>

**开始时宣布：** "我正在使用AI部门专属需求验证技能来验证AI产品需求。"

## 何时使用

**在以下情况使用：**
- AI PRD编写完成后，在分享给开发团队之前
- AI实施开始前，确保需求完整且工程可行
- 收集到新AI需求后，验证其质量和可行性
- AI产品规划会议中，评估需求准备度

**在以下情况不使用：**
- 技术实施细节讨论
- 代码审查
- 纯技术架构决策

## AI产品验证核心维度

### 一、四大确定性边界验证（所有AI需求强制，一票否决）

这是需求收集阶段确认，PRD阶段固化的核心工程契约。必须逐项验证：

| 检查项 | 验证标准 | 状态 |
|--------|----------|------|
| **✅ 确定性边界** | 是否明确区分确定性硬编码 vs AI推理？重试策略是否定义？重试失败后的降级UI是否明确？ | [ ] 通过 |
| **✅ Context资源管理** | 是否明确模型上下文窗口大小？超窗口时的数据截断规则是否定义？（按时间/摘要/核心过滤/分块） | [ ] 通过 |
| **✅ Golden Dataset** | 是否提供覆盖核心场景的Golden Dataset？测试集路径是否记录？通过率要求是否明确？ | [ ] 通过 |
| **✅ 文档版本化** | PRD是否明确将存放在项目 `/docs/`，并通过Git版本管理？ | [ ] 通过 |

> ⚠️ **一票否决规则：** 四项中有任何一项不通过 → 需求验证不通过，不能进入下一阶段。这是工程不可行伪需求的核心过滤器。

### 四、人机协作与可解释性验证（AI产品特有）

| 检查项 | 验证标准 | 状态 |

| AI职责是否明确？ | AI负责哪些部分（规则明确、数据密集、重复性工作）？ | [ ] 通过 |
| 人类职责是否明确？ | 哪些场景必须人类介入（情感沟通、复杂决策、高风险操作）？ | [ ] 通过 |

**参考模板：**

| 场景类型 | AI负责部分 | 人类负责部分 | 切换触发条件 |

### 七、风险与兜底验证

AI产品必须有刹车系统，验证风险和兜底是否到位，同时权责前置清晰切分归因：

| 风险ID | 描述 | 是否识别 | 是否有缓解策略 | 预期归因 | 状态 |
|--------|------|----------|----------------|----------|------|
| R001 | 模型置信度低 → 自动转人工 | [ ] | [ ] | 算法/策略 | [ ] 通过 |
| R002 | 模型输出格式错误 → 重试+降级 | [ ] | [ ] | 算法/策略 | [ ] 通过 |
| R003 | 数据质量差 → 紧急标注+回退 | [ ] | [ ] | 数据/算法 | [ ] 通过 |
| R004 | 生成有害内容 → 审核过滤 | [ ] | [ ] | 工程/产品 | [ ] 通过 |
| R005 | Token成本超预算 → 月度封顶+切小模型 | [ ] | [ ] | 产品/工程 | [ ] 通过 |
| R006 | 用户输入违规 → 拒绝响应 | [ ] | [ ] | 用户侧 | [ ] 通过 |
| R007 | API调用超时 → 降级返回 | [ ] | [ ] | 工程侧 | [ ] 通过 |

**权责划分原则：<sup>[2]</sup>**
- **用户侧输入异常**：用户输入不合规、提示词违背常识 → 不归因产品/算法
- **工程链路故障**：API调用超时、系统错误 → 归因工程团队
- **算法/策略缺陷**：输入标准、链路正常但输出离谱 → 归因算法/策略团队

### 八、成功指标验证

遵循SMART-R原则从业务目标逐层拆解到技术指标<sup>[5]</sup>，验证三层指标是否完整：

| 层级 | 验证内容 | 状态 |
|------|----------|------|
| **业务指标** | 成本节约/效率提升/用户满意度是否量化？ | [ ] 通过 |
| **产品指标** | 功能使用率/人工介入率/任务完成率是否定义？ | [ ] 通过 |
| **技术指标** | 模型准确率/TTFT/SLA是否明确？ | [ ] 通过 |

**SMART-R原则回顾：**
- **S**pecific：明确场景与对象
- **M**easurable：定义量化指标
- **A**chievable：结合数据质量与算力评估可行性
- **R**elevant：指标与核心业务目标对齐
- **T**ime-bound：设定迭代周期与上线节点
- **R**eviewable：建立指标漂移检测与归因分析机制

### 九、Golden Dataset验收标准验证

验收AI产品必须用全量化测试集说话，不能靠主观感觉<sup>[2]</sup>。验证：

| 检查项 | 验证标准 | 状态 |
|--------|----------|------|
| 测试集构成是否符合黄金比例？ | 70%常规正向case + 20%边界合规case + 10%异常负向case | [ ] 通过 |
| 测试集大小是否足够？ | 核心场景≥20个，复杂场景建议≥100个 | [ ] 通过 |
| 测试集是否提前固化？ | 需求评审阶段已确认，验收阶段不新增/修改 | [ ] 通过 |
| 通过率及格线是否明确？ | 最低通过率要求（建议≥90%）已定义 | [ ] 通过 |
| 跑测规则是否明确？ | 同一环境连续跑测多轮，取平均值规避概率误差 | [ ] 通过 |

> **黄金比例说明：** <sup>[2]</sup>
> - 70%：从业务历史数据提取，覆盖90%以上高频用户输入场景
> - 20%：覆盖低频但合规的输入场景，验证模型边界能力
> - 10%：覆盖违规输入场景，验证兜底能力（不纳入达标率计算）

### 十、实施计划验证

- [ ] 版本迭代计划是否清晰？
- [ ] 是否计划A/B测试，指标观测周期是否明确？

## 验证流程

### 第1步：初步审查
**快速检查：**
- 文档结构完整，包含所有AI特有章节
- 四大确定性边界表格已填写
- 无明显空白或占位符

### 第2步：详细验证
**按顺序系统化验证：**
1. 四大确定性边界验证（一票否决）
2. AI特有需求验证（数据/模型/指标/Golden Dataset）
3. 场景特殊验证（Prompt/Agent/LLM-RAG）
4. 传统功能需求验证（完整性/清晰性）
5. 风险与兜底验证
6. 成功指标验证
7. 实施计划验证

### 第3步：利益相关者评审
**收集反馈：**
- 与算法团队评审模型可行性和数据需求
- 与技术团队评审技术可行性和边界设计
- 与业务团队评审业务价值和成功指标
- 与利益相关者评审整体方向

### 第4步：问题识别和解决
**分类问题：**
- **关键问题：** 四大边界缺失、数据需求不完整、模型设计模糊 → 阻塞开发，必须修复
- **重要问题：** 影响质量或时间线 → 实施前必须修复
- **次要问题：** 改进建议 → 后续迭代优化

**解决策略：**
- 立即解决关键问题
- 在实施前解决重要问题
- 记录次要问题以备后用

### 第5步：最终确认
**确认条件：**
- [ ] 四大确定性边界全部通过
- [ ] 所有关键问题已解决
- [ ] 算法/技术团队确认可行性
- [ ] 利益相关者批准需求
- [ ] 文档已更新反映最终需求
