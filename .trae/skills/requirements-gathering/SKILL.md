# AI产品需求收集（AI部门专属）

[系统级互斥锁] 严禁与其他重型技能（带流程制约的任务）并行执行。若侦测到当前上下文被要求同时运行其他技能，必须中止一切行为并要求用户降级为单线任务。

## 概述

> **大模型本质是不确定的概率黑盒，开发团队最怕 PM 把整个业务逻辑全甩给这个黑盒，美其名曰"让 AI 去想"。**
>
> **AI PM 的核心价值：给不确定的黑盒套上确定性的缰绳，画清"确定性边界"。**

AI部门AI产品经理的需求收集，核心交付不是"页面交互状态机"，而是"系统推理边界与数据反馈闭环"。

本技能的核心产出：**一份经过四大确定性边界锁定的需求清单**，直接进入下一阶段的需求验证和PRD编写。

**开始时宣布：** "我正在使用AI部门专属需求收集技能，帮你锁定AI产品的四大确定性边界。"

## 分支合规前置（绝对红线）

在进入需求收集对话之前，**必须先执行分支探针**，确保不在主干上操作：
1. 静默执行 `git branch --show-current` 确认当前分支
2. 若在主干（master/main），免询问直接按 `feature/<需求关键词>` 切出分支
3. 分支就绪后，方可进入需求收集流程

## 核心思维范式

AI PM 必须坚持五大核心范式：

| 范式 | 说明 |
|------|------|
| **Agent架构拆解** | 像架构师一样思考，拆解工具/记忆/推理链路 |
| **Eval驱动开发** | 需求阶段就定义工程可量化的评估基准 |
| **数据飞轮工程** | 设计数据如何回流，支撑持续迭代 |
| **Prompt架构前置** | 需求阶段就定义System角色"灵魂" |
| **边界死磕** | 明确界定确定性逻辑和AI推理边界 |
| **Context即资源** | 把上下文窗口当核心资源管理 |
| **测试集即PRD** | 结构化Golden Dataset胜过千言万语 |
| **Git版本化** | 需求存在 `/docs/`，随代码一起迭代 |
| **数据优先** | 实现需求需要什么数据 |
| **风险可控** | 接受不完美，用低成本兜底 |
| **价值桥梁** | 向各方翻译价值与可行性 |
| **持续迭代** | 用数据飞轮不断优化效果 |
| **数据+AI双引擎** | 用AI放大洞察能力 |
| **JTBD任务思维** | 用户雇AI完成什么任务 |

## 四大硬核工程原则（必须遵守）

### 1. 死磕"确定性"与"非确定性"的交界

AI PM 最大的灾难，就是分不清哪些该写死，哪些该交给模型。

| 反面教材（甩锅） | 专业输出（工程友好） |
|------------------|----------------------|
| "系统要智能提取用户的报销信息并入库" | 上游：明确格式清洗规则（JPG/PDF/PNG支持，统一DNN预处理）<br>下游：定义严格JSON Schema输出（`amount: number`/`date: string`）<br>异常：格式错误重试2次 → 仍失败展示"请上传清晰照片"降级UI |

**必须明确回答：**
- 哪部分逻辑是**确定性硬编码**（格式清洗、Schema校验）？
- 哪部分逻辑**交给AI推理**？
- 如果AI输出格式错乱/缺字段，**重试策略**是什么？
- 重试失败后，**降级UI**如何展示？

### 2. 把"上下文窗口"当成核心资源来管理

不管本地跑多强算力的服务器，显存和上下文都是有极限的。**PM 必须像传统工程师懂内存管理一样懂 Context 管理**。

**需求必须说明：**
- 当前选用模型的上下文窗口是多少（32K/64K/128K）？
- 当历史记录/召回文本超过限制，**数据截断规则**是什么？
  - 按时间截断（保留最新）？
  - 触发一次总结摘要压缩？
  - 通过严格规则只保留核心关联字段？

**强制要求：** 在写业务流时，必须把数据截断规则作为前置条件写清楚，**不能等测试环境 OOM 了才补救**。

### 3. 测试集（Golden Dataset）就是真正的 PRD

在 AI 专项开发中，长篇大论的文字需求远不如结构化的测试用例有用。

- ❌ **不及格：** 只有文字描述，没有测试数据
- ✅ **合格：** 交付包含结构化测试集（≥20个高频场景的 Input + 期望 Output）
- ✅ **工程价值：** 这些数据可以直接转化为自动化测试用例，集成到 CI/CD。PR 合并导致核心测试集通过率下降 → 直接打回。用工程标准倒逼需求质量。

### 4. 彻底改变文档的存在方式

AI 迭代太快，传统"发版一份 Word 文档"早就行不通了。

- ✅ **存放位置：** 需求说明和规则沉淀必须是 Markdown，作为 SSOT（唯一事实来源）严格存放在项目 `/docs/` 目录
- ✅ **版本管理：** PM 需求调整必须和代码一样，通过 Git 提交流程，每一次规则修改、每一个 Prompt 调整都和代码版本强绑定、可追溯

> **工程格言：** 那些虚无缥缈的"智能感"，最终都要靠最严谨的结构化字段、兜底逻辑和版本控制来托底。

## JTBD核心追问：用户雇AI完成什么任务

> **用户根本不是"买AI"，而是"雇AI"来搞定特定场景下的棘手任务！**

永远追问：
- 用户在什么**情境**下需要AI？
- 用户想让AI帮他完成什么**任务**？
- 完成任务后能帮用户获得什么**进步/价值**？

**避免伪需求陷阱：**
- ❌ 炫技陷阱：AI能生成万字文，但用户真的需要万字文吗？
- ❌ 虚荣指标：DAU/MAU很高，但任务完成率有多少？
- ✅ 聚焦任务：用户需要的是"能带货的商品详情页"，不是"一万个字"

**任务完成度四维度：**
1. 任务成功率 → 能帮用户完成目标吗？
2. 任务完成时间 → 比传统方案快多少？
3. 任务复杂度边界 → 能处理哪些复杂场景？
4. 用户满意度 → 使用过程体验好不好？

完整JTBD方法论请阅读：`references/ai-product-special.md`

## AI需求四步挖掘框架

### 第一步：场景筛选 → 三维筛选法锁定高价值场景

优先聚焦**高频、高痛、高价值**场景，避免"为了AI而AI"：

| 维度 | 判断标准 |
|------|----------|
| **痛点强度** | 用户是否愿意付费/花时间？不解决是否流失/损失？ |
| **发生频率** | 每天/每周发生？高频才能覆盖AI研发成本 |
| **价值量化** | 能否明确降本X%/增收Y%/节省Z分钟？ |

**输出分类：**
- ✅ **优先聚焦**：三高场景（高频+高痛+高价值）
- ⚠️ **谨慎评估**：有一项不满足，需要验证
- ❌ **暂不考虑**：低频低痛低价值，先不做

### 第二步：需求验证 → 三招辨真伪

验证需求真实性，避免伪需求陷阱：

1. **用户深度访谈** → 听真实反馈，不只看表面需求
   - 不仅问"你需要什么"，更问"你当前如何解决""最重要的是什么""能接受什么成本"
   - 关注用户的犹豫和吐槽，背后就是隐性需求
   - 经典引导问题："如果AI能帮你多做一件事，你希望是什么？"

2. **MVP低成本验证** → 用人工/Excel替代验证，再用AI提效
   - 示例：想做"AI简历优化"，先做人工修改服务验证付费意愿
   - 如果用户不愿意付费，说明需求是伪需求

3. **行为数据佐证** → 用现有数据验证，不只信口头反馈
   - 分析用户行为轨迹，看痛点是否真实存在

### 第三步：需求拆解 → 模糊期望 → 可落地需求

AI产品收集到的需求往往是模糊的（如"AI更智能""用着更方便"），必须拆解为具体可量化的需求点：

1. **筛选核心需求** → 保留高频刚需，砍掉非核心
2. **拆解细节** → 每个需求点明确：格式支持、输出要求、性能指标、时间限制
3. **关联用户价值** → 每个需求点明确解决什么问题

> 关键原则：拆解必须结合AI技术特性，避免拆解出技术无法实现的需求

### 第四步：技术边界评估 → 能做/不能做/怎么做

这是AI产品和传统产品最大的区别！需求拆解后必须评估技术边界：

**四个评估维度：**
1. **大模型能力**：当前模型能否实现？
2. **算力成本**：Token成本是否超出预算？
3. **开发周期**：是否符合迭代节奏？
4. **技术风险**：是否存在未解决的技术瓶颈？

**分类处理：**
- ✅ **可直接实现** → 纳入当前迭代
- 🛠 **需优化简化** → 调整需求范围（如"识别所有格式"→"识别常见Word/PDF"）
- ⏸ **暂不实现** → 放入需求池，等技术成熟

## AI产品必做检查清单

### 核心六问（必做）

**1. 数据基础**
- 实现该AI能力需要哪些数据？
- 数据来源是否可靠？质量能否达标？
- 是否需要持续收集用户反馈来优化？

**2. 能力边界**
- AI必须做到什么程度算满足需求？（准确率/召回率目标）
- AI做不到什么是可以接受的？
- 如果AI输出不确定，是否需要人工兜底？如何兜底？

**3. 成本约束**
- Token成本预算是多少？
- 可接受的最大响应延迟是多少？
- 是否需要分层策略（简单问题用小模型，复杂问题用大模型）？

**4. 反馈机制**
- 如何收集用户对AI输出的反馈？
- 是否需要"点赞/点踩"按钮？
- 反馈数据如何用于模型迭代？

**5. 幻觉控制**
- AI产生幻觉（错误输出）会造成什么业务风险？
- 有什么机制可以降低幻觉影响？
- 是否需要引用来源标注，方便用户验证？

**6. 迭代路径**
- 第一版可接受的最低效果标准是什么？
- 是否分阶段（先MVP验证，再效果调优）？

**7. 合规与隐私**
- 数据采集是否合法？是否获得用户授权？
- 是否涉及敏感数据（人脸/语音/财务）？需要如何脱敏？
- 是否存在算法偏见风险？需要做哪些排查？

### 四大边界确认（所有AI需求强制）

> **核心原则：大模型是不确定的概率黑盒，AI PM 必须给它套上确定性的缰绳。**

| 检查项 | 确认内容 |
|--------|----------|
| **✅ 确定性边界** | 是否明确界定了哪部分逻辑是确定性硬编码（格式清洗、Schema校验），哪部分交给AI推理？如果AI输出格式错误，重试策略和降级UI是什么？ |
| **✅ Context资源管理** | 如果上下文超过模型窗口限制（32K/128K），数据截断规则是什么？按时间截断/摘要压缩/核心字段保留？ |
| **✅ Golden Dataset** | 是否提供了覆盖核心场景的结构化测试集（Input → Expected Output）？这些数据可直接用于CI/CD自动化测试。 |
| **✅ 文档版本化** | 需求是否以Markdown存放在项目 `/docs/`，并通过Git版本控制，和代码强绑定可追溯？ |

### Agent 架构与评测三问（进阶强管控，Agent需求强制）

**凡是涉及复杂任务处理的 AI 需求（Agent、智能助手、自动化工作流），必须强制完成以下确认，并写入 `/docs/` 作为唯一事实来源（SSOT）：**

**最佳实践：** 传统用户故事"用户要什么"不够，Agent需求要写 **Agent Story** →"Agent怎么干"<sup>[6]</sup>。

**1. 智能体工作流 (Agentic Workflow) 拆解：**
- **Agent Story：** 把"用户要什么"改写为"Agent如何一步步完成任务"
- **七步交互闭环：** 听(接收)→懂(意图识别)→补(追问信息)→做(调用工具)→想(决策)→说(回复)→学(记录)<sup>[6]</sup>
- **工具授权：** 列出该功能所需的所有外部 API、数据库读写权限、可调用的自动化脚本清单。
- **记忆机制：** 明确短期上下文窗口的需求，以及是否需要通过 RAG 或向量数据库建立长期记忆（如用户偏好、个性化知识库）。
- **人机协同边界：** 在哪个决策节点必须设置"人工确认（Human-in-the-loop）"？一步到位还是先出计划再确认执行？

**2. 评估基准前置 (Eval-Driven)：**
- **黄金数据集：** 业务方是否已提供覆盖核心场景的 `Input -> Expected Output` 测试集？（如果没有，需求不予准入）
- **评测维度定义：** 明确输出质量的验收权重（如：事实准确度 40%、格式合规性 40%、语气亲和度 20%），定义 LLM-as-a-Judge 的打分规则。

**3. 延迟与数据飞轮 (Ops & Flywheel)：**
- **延迟容忍度：** 明确首字输出时间 (TTFT) 约束。在端侧/本地模型高负载或云端超时下的异步降级体验是什么？
- **数据回流闭环：** 当输出不符合预期时，用户的纠错行为（编辑、重试、否定、点踩）如何以结构化数据落表，以便后续微调 (Fine-tuning) 或优化 Prompt？

> **强制规则：** 在需求阶段没有完成边界确认的AI需求，不得进入下一环节。这是过滤工程不可实现伪需求的关键闸门。
>
> **最佳实践：** LLM/Agent产品别搞瀑布流，**Demo驱动开发** → 快速做Demo拿反馈，比写100页PRD管用<sup>[6]</sup>。

## AI辅助需求收集工具用法

大模型本身可以大幅提升需求收集效率：

### 1. 零散需求结构化整理
- 输入：所有聊天记录、会议纪要、录音转文字
- 指令："请将以下零散需求整理为规范文档，包含背景、功能需求、非功能需求、边界条件"
- 收益：2天工作量 → 30分钟完成初稿

### 2. 模糊需求自动拆解
- 输入："我们需要做一个AI驱动的智能客服系统"
- 指令："请拆解这个模糊需求，生成功能点清单（覆盖用户/客服/管理侧）和澄清问题清单"
- 收益：一次性列出所有需要确认的细节，避免反复沟通

### 3. 用户评论痛点挖掘
- 输入：10万条用户评论/客服对话
- 指令："请分析这些评论，聚类生成核心痛点清单，按频率排序"
- 收益：AI自动挖掘隐性需求，比人工梳理快10倍

## 伪AI需求识别

在需求收集阶段就要识别并排除不可行的伪需求：

| 伪需求特征 | 应对策略 |
|------------|----------|
| **问而不能答** → 问题超出当前AI能力 | 重新定义问题，找AI能解决的子问题 |
| **答而不可控** → 输出不稳定，风险太高 | 增加人工兜底环节，降低风险 |
| **控而无反馈** → 无法收集反馈，无法迭代 | 设计反馈机制，或放弃该需求 |
