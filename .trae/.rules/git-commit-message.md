# 提交信息生成规范

## 格式

```
[类型](模块): 动作及意图
```

- **类型**：必须从下方字典选取，方括号包裹
- **模块**：非空的中文或英文，圆括号包裹
- **动作及意图**：动词开头，说清改了什么、为什么，**禁止 `update/fix/修改` 等模糊语义**（commit-msg hook 会拦）

## 类型字典（单一事实源，与 commit-msg hook `check_msg_format` 一致）

| 标识 | 用途 |
|------|------|
| `[文档]` | 新增/更新 PRD、ADR 等需求/设计文档 |
| `[资产]` | 原型、UI 规范、流程图、图片等资产更新 |
| `[修正]` | 修复文档语病、排版、逻辑错漏（文档类修正）|
| `[feat]` | 新功能（代码）|
| `[fix]` | 修复 bug（代码）|
| `[refactor]` | 重构（不改外部行为）|
| `[chore]` | 杂务/构建配置/依赖等非功能性改动 |
| `[docs]` | 代码项目的文档变更 |
| `[style]` | 格式调整（空格/分号等，不改逻辑）|
| `[test]` | 新增/修改测试 |
| `[build]` | 构建系统或外部依赖变更 |
| `[ci]` | CI 配置/脚本变更 |

## 示例

```
[文档](用户系统): 新增 v2.0 用户权限模块 PRD
[feat](登录): 添加手机号验证码登录功能
[fix](支付): 修复金额计算精度丢失问题
[refactor](数据层): 抽取缓存逻辑为独立模块
```

## 红线（commit-msg hook 会硬拦，生成时就要避开）

- ❌ 空信息 / 纯数字 / 纯标点
- ❌ `update`、`fix`、`修改` 等无信息量模糊词（仅写这些会被 `check_msg_no_vague` 拦）
- ❌ 不符合 `[类型](模块): 描述` 格式
- ✅ 合并提交（`Merge branch ...`）和 revert（`Revert "..."`）由 git 自动生成，hook 放行，无需套格式
