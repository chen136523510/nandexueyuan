# 文档管理体系（强制执行）

> 状态：Accepted | 决策人：陈梓键 | 日期：2026-07-17

---

## 1. 核心原则

**文档先行**。任何需求开发前，必须先有对应的需求文档/调研文档。代码改了但文档没更新 = 任务未完成。

---

## 2. 四类文档

| 类型 | 位置 | 用途 | 生命周期 |
|------|------|------|------|
| **需求池** | `prd/01-需求文档/00-基础数据/需求池.md` | 待开发清单，指导开发优先级 | 做完从"待开发"移至"已完成" |
| **需求文档** | 各模块 `01-需求/` | 归档模块实现逻辑、业务契约 | 长期保留 |
| **changelog / bug-log** | 各模块目录下 | 变更回溯、bug 追踪 | 长期保留 |
| **调研文档** | 分两级（见下） | 技术调研、方案选型 | 长期保留 |

---

## 3. 调研文档分级

| 级别 | 位置 | 适用场景 | 示例 |
|------|------|------|------|
| **全局调研** | `prd/01-需求文档/00-调研/` | 跨模块、项目结构、部署策略、工具链、文档管理、非产品本身的调研 | ComfyUI 工作流、部署方案、Git 策略 |
| **模块调研** | 各模块 `03-调研/` | 仅限该模块的技术选型、实现方案 | NPC AI 对话接入、泰拉瑞亚模块研究 |

**判断标准**：如果调研内容涉及多个模块或项目本身的基础设施（工具链、部署、Git），放全局调研；如果只服务单个模块，放模块调研。

---

## 4. 需求池管理

### 4.1 格式

```markdown
## 待开发

| 编号 | 需求 | 模块 | 优先级 | 关联文档 | 备注 |
|------|------|------|:--:|------|------|
| R-XXX | 需求描述 | 模块名 | P1/P2 | [文档名](路径) | 备注 |

## 已完成

| 编号 | 需求 | 模块 | 完成日期 | 关联文档 | Changelog |
|------|------|------|------|------|------|
| F-XXX | 需求描述 | 模块名 | YYYY-MM-DD | [文档名](路径) | [changelog](路径) |
```

### 4.2 编号规则

- `R-XXX`：需求池中的需求（Requirement），3 位数字
- `F-XXX`：MVP 需求文档中的功能编号（Feature），与需求池中的 R 编号对应
- 已完成的 `R-XXX` 移至"已完成"区，改用对应的 `F-XXX` 编号记录

### 4.3 操作规则

- 新需求加入"待开发"区
- 开发完成后，从"待开发"移至"已完成"
- "已完成"区保留历史记录，不清除
- 每次移动需同步更新对应模块的 changelog

---

## 5. 文档更新检查清单

完成任何代码修改后，必须逐项确认：

```
[ ] 1. 需求池是否已更新？（待开发 -> 已完成）
[ ] 2. changelog 是否已记录？
[ ] 3. 如果是 Bug 修复，bug-log 是否已登记？
[ ] 4. 如果涉及功能需求，需求文档是否已同步？
[ ] 5. 如果涉及架构决策，是否需要新增 ADR？
[ ] 6. 调研文档是否放对了位置（全局/模块）？
```

```
    ...
  01-首页/

  03-男德通/
  04-德塔/
    01-需求/              # 德塔模块需求文档
    02-设计/              # 德塔模块设计文档
    03-调研/              # 德塔模块级调研
    04-技术方案/
    05-接口契约/

    bug-log.md            # 德塔模块 bug 记录
```
