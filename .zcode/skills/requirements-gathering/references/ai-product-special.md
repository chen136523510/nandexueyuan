# AI产品需求收集方法论

AI产品（大模型驱动、生成式AI、智能Agent等）需求收集必须遵循特殊方法论。

## 核心挑战

AI产品需求收集的核心难点：
- 用户并不真正了解AI的"可能性边界"，只会表达痛点，不会描述可落地的功能
- AI能力和用户意图之间存在高度**非对称性认知差异**
- AI系统是概率性系统，不是固定规则执行，无法精确枚举所有场景
- 超70%的AI项目无法规模化落地，根源都是需求找错了场景 <sup>[1](#ref1)</sup>

## JTBD思维：用户雇AI完成什么任务

> "用户根本不是'买AI'，而是'雇AI'来搞定特定场景下的棘手任务！" <sup>[1](#ref1)</sup>

### 为什么JTBD是AI产品的元认知工具

AI存在本质悖论：**能力越强，定位越模糊**。当大模型什么都能做时，用户反而不知道用它来做什么。JTBD通过"情境-任务-目标"三元结构，为这种模糊性提供清晰解题思路。

### 任务完成度指标（替代虚荣指标）

| 指标 | 说明 | 示例 |
|------|------|------|
| **任务成功率** | 用户雇佣AI后是否实现预期目标 | AI简历优化：优化后的简历获得面试邀请的比例（≠简历修改次数） |
| **任务完成时间** | 相比传统方案，效率提升多少 | AI财务报销：从2小时压缩至15分钟 |
| **任务复杂度边界** | AI能处理哪些复杂场景 | AI法律咨询：不仅解决常见问题，还能处理跨境电商合规 |
| **用户满意度** | 任务完成过程的情感体验 | AI隐私保护：用户无需担心数据泄露 |

### 经典案例：钉钉AI的JTBD转型

**传统CRM模式**：卖许可证，按用户数收费，提供功能，不保证结果 → 用户付费后仍需手动干活

**JTBD新模式**：卖"客户跟进效率提升"结果，按成功跟进商机数收费 → 钉钉AI自动抓需求、生成话术、提醒节点，直接对商机转化负责 → **20%+企业付费转化率** <sup>[1](#ref1)</sup>

## AI产品需求三大核心特征

| 特征 | 说明 | 应对策略 |
|------|------|----------|
| **开放性** | 输入输出往往不唯一，用户期待"灵活响应"而非严格定义的返回格式 | 定义能力边界而非固定逻辑，接受多解空间 |
| **语义性** | 需求中大量包含"风格""质量""可读性""像人一样"等无法量化但期望明确的描述 | 必须找到可落地的评估标准和样例 |
| **反馈性** | 很多AI系统依赖多轮优化、用户反馈、后处理纠正，需要构建效果闭环 | 设计反馈收集机制和数据飞轮 |

## AI产品经理核心思维

| 思维 | 说明 |
|------|------|
| **数据优先** | 实现需求需要什么数据 |
| **风险可控** | 接受不完美，用低成本方案兜底 |
| **价值桥梁** | 向各方翻译价值和技术可行性 |
| **持续迭代** | 用数据飞轮不断优化效果 |
| **数据+AI双引擎** | 用AI放大洞察能力 |

## AI产品专属需求收集方法

### 1. 场景解构法（定性研究）

**核心：找到AI技术可以降维打击的"高能耗"场景**

提问框架（能量消耗审问）：
- 用户当前工作流中，哪个环节最"耗费心神"？
- 这个环节重复劳动比例有多高？
- 如果AI能自动化这个环节，能节省多少时间/成本？
- 用户愿意为什么样的效果付费？

**操作步骤：**
1. 绘制完整用户工作流
2. 给每个环节标注"体力消耗"（重复劳动）和"脑力消耗"（决策难度）
3. 聚焦"高能耗"环节，用AI替代或增强
4. 量化价值，戳破"我想要"的虚假繁荣

### 2. 技术联想坊（焦点小组）

**核心：激发用户将技术可能性与实际场景结合**

操作方法：
- 通过演示Sora、Runway、Pika等先进AI能力，打开用户想象力
- 引导提问："如果AI能做到这个程度，你认为在咱们业务中能用来解决什么问题？"
- 记录跨界联想，从中发现创新应用场景

### 3. 人机对抗行为分析（观察法）

**核心：观察用户如何"驯服"AI，找到产品切入点**

需要记录的关键指标：
- **放弃阈值**：用户尝试几次后会放弃使用AI
- **调教行为**：用户是否在反复调整Prompt获得更好结果
- **信任拐点**：用户什么时候开始信任AI输出，愿意直接使用

**案例：** ArcadeSoftware在用户完成创作、最想分享成果的"信任拐点"上弹出付费窗口，实现了精准的用户转化。

### 4. 数据需求五步法（数据优先思维）

AI产品必须先想数据，再想功能：

| 步骤 | 提问要点 |
|------|----------|
| 1. 明确能力目标 | AI具体要做什么？（如智能客服要"听懂用户意图"） |
| 2. 拆解数据需求 | 需要哪些数据才能实现这个能力？ |
| 3. 定义质量标准 | 文字准确率要超过多少？覆盖多少用户意图？ |
| 4. 确认数据来源 | 多少来自真实数据？多少需要人工标注？ |
| 5. 设计更新机制 | 数据如何持续迭代，形成数据飞轮？ |

**数据飞轮设计要点：**
- 加入用户反馈按钮（通常约8%的用户会点击反馈）
- 对AI输出进行人工标注，反哺模型训练
- 用的人越多，数据越多，效果越好，形成良性循环

### 5. 需求转化三角法（跨角色沟通）

AI产品经理需要成为"三维翻译官"：

| 沟通对象 | 沟通内容 |
|----------|----------|
| **老板/决策者** | 讲投入产出表（花多少钱、能赚多少、多久回本） |
| **AI算法与应用工程师** | 讲业务目标对应的技术指标（错误识别率提15%、漏检率低于3%） |
| **用户** | 讲AI能做啥、不能做啥，设置合理预期 |

### 6. 数据+AI双引擎驱动需求挖掘

传统人工需求挖掘依赖主观访谈，样本小、反馈滞后，难以适应Z世代碎片化需求 <sup>[2](#ref2)</sup>。

**AI需求雷达模型工作流：**

1. **数据抓取** → 八爪鱼爬虫从社交媒体、应用商店、客服对话抓取海量用户生成内容
2. **情感分析** → GPT-4对抓取文本做情感分析，判断用户对产品的情感倾向
3. **聚类发现** → Python聚类分析发现需求类别与模式
4. **反常识验证** → 对比用户"说的"和"做的"，发现真实需求

**实战案例：** 某教育App团队抓取B站"学习打卡"弹幕，通过AI分析发现"社恐式学习"需求，推出"无声自习室"功能，次日留存率提升22% <sup>[2](#ref2)</sup>。

### 7. AI辅助场景分析四步法

AI能帮我们从"猜测用户需求"升级到"洞察用户本质"，拓展广度、提升深度、加速流程 <sup>[3](#ref3)</sup>。

| 步骤 | 目标 | AI用法 |
|------|------|--------|
| **1. 场景挖掘与假设生成** | 生成潜在场景 | 输入背景，让AI生成10个潜在场景（包含用户角色、情景、目标、痛点） |
| **2. 场景细化与要素拆解** | 深入理解细节 | 按"场景五要素"（用户、环境、动机、行为、痛点）结构化拆解 |
| **3. 用户旅程图绘制** | 发现体验痛点 | AI快速生成用户旅程图（行为、想法、情绪、痛点） |
| **4. 需求提炼与功能假设** | 转化为产品方案 | 基于痛点列出解决方案，评估实现成本和预期效果 |

**关键原则：** AI定位是"初级顾问"和"效率加速器"，不是决策者。产品经理依然对最终结果负责。保持批判性思维，不断追问："这个结论符合逻辑吗？有数据支撑吗？"

## 伪AI需求识别

在需求收集阶段就要识别并排除不可行的伪需求：

| 伪需求特征 | 说明 | 应对策略 |
|------------|------|----------|
| **问而不能答** | 问题超出当前AI能力边界 | 重新定义问题，找AI能解决的子问题 |
| **答而不可控** | AI输出无法保证稳定性，业务风险太高 | 增加人工兜底环节，降低风险 |
| **控而无反馈** | 无法收集用户反馈，无法迭代优化 | 设计反馈机制，或放弃该需求 |

## AI需求验收特殊要点

- 接受"部分准确"，但必须明确准确率底线和错误容忍度
- 设计人工兜底方案（置信度低于阈值转人工）
- 明确反馈收集机制，支持持续优化
- Token成本和延迟也是非功能需求的重要组成

## AI产品需求补充6问模板

当收集AI产品需求时，在完成通用12问后，必须补充以下6问：

```markdown
## AI需求补充6问记录

### 必做6问
1. 数据基础：[数据来源、质量、更新机制]
2. 能力边界：[准确率目标、可接受误差、人工兜底方案]
3. 成本约束：[Token预算、延迟要求、分层策略]
4. 反馈机制：[反馈收集方式、数据迭代流程]
5. 幻觉控制：[风险评估、幻觉降低机制]
6. 迭代路径：[MVP标准、分阶段计划]
```
