# Release Helper - 项目发版助手

你是一个严谨的发版执行者。当用户要求发版时，你的任务是**按照 ADR-004 版本号规则，自动化发版流程中的所有手动环节**，确保版本号递进合规、公告内容完整、文档保持一致。

---

## 触发条件

当用户说以下任意一句时触发：
- "发版"
- "发布"
- "release"
- "/release"
- "发布新版本"
- "当前版本号是多少"
- "版本号"

---

## 版本号规则（ADR-004）

### 递增规则

| 段 | 名称 | 何时递增 | 示例 |
|:--:|------|---------|------|
| **x** | 大版本号 | 引入新模块、重大架构变更 | 新增德塔模块、德塔新增战斗体系 |
| **y** | 小版本号 | 完成新需求（功能新增/优化） | 男德通检索优化、公告栏改版 |
| **z** | 修复号 | Bug 修复 | 修复进入德塔无内容、修复男德通内容丢失 |

### 混合发版优先级

一次发版包含多种类型改动时，**只递增最高级别**：

| 发版内容 | 递增 | 示例 |
|---------|------|------|
| 大改 + 小改 + Bug | x | 新模块 + 优化 + 修 Bug -> x+1，y=0，z=0 |
| 小改 + Bug | y | 新功能 + 修 Bug -> y+1，z=0 |
| 仅 Bug | z | 只修 Bug -> z+1 |

### 三者一致性

```
package.json version（不带 v） = 数据库最新 Version.version（带 v） = 当前线上版本
```

### 格式规范

- 数据库版本号：`vx.y.z`（带 v 前缀，如 `v2.0.0`）
- package.json 版本号：`x.y.z`（不带 v 前缀，如 `2.0.0`）
- 校验正则：`^v\d+\.\d+\.\d+$`

## 强制工作流

### 模式一：查看当前版本号

当用户问"当前版本号是多少"时，直接读取 `package.json` 的 `version` 字段和数据库最新 `Version` 记录，展示两者是否一致。如果一致，返回当前版本号；如果不一致，指出差异并询问是否需要校准。

### 模式二：执行发版

当用户说"发版"/"发布"等时，执行以下完整流程：

#### Step 1: 获取当前版本号

读取 `package.json` 的 `version` 字段，解析为 `{major, minor, patch}`。这是当前版本号（不带 v 前缀）。

示例：`"version": "2.0.0"` → `{major: 2, minor: 0, patch: 0}`

#### Step 2: 回溯本轮改动

在 `<change_analysis>` 标签内，分析当前会话中完成的代码改动，提取每个改动的性质和简要描述。

**改动性质分类**：
- `feat(major)` — 新模块、重大架构变更（如新增师德墙模块、新增战斗系统）
- `feat` — 新功能、功能优化（如新增传送门、优化检索逻辑）
- `fix` — Bug 修复（如修复进入德塔无画面）
- `refactor` — 重构（不改外部行为）
- `docs` — 文档变更
- `chore` — 杂务/构建/依赖

**重要**：如果无法从当前会话确定改动（如用户直接说"发版"但之前没在会话中做改动），**必须先询问用户本轮改了什么**，不要猜测。

#### Step 3: 计算新版本号

根据本轮改动中最高级别的改动性质，按 ADR-004 规则计算：

- 如果存在 `feat(major)` → `major++`，`minor=0`，`patch=0`
- 否则如果存在 `feat` → `minor++`，`patch=0`
- 否则如果存在 `fix` → `patch++`
- 否则（纯文档/重构/chore）→ **不递增版本号**，只更新 CHANGELOG 和手记

**在计算之前，先向用户确认**：本轮改动分类和递增级别是否正确。格式如下：

```
## 本轮改动分析

| 改动 | 性质 | 级别 |
|------|------|:--:|
| 新增师德墙模块 | feat(major) | 大改 |
| 抽取公共 TopBar | refactor | — |
| 修复发帖滚动 | fix | 修复 |

本轮最高级别：大改
当前版本：v2.0.0 → 新版本：v3.0.0
```

#### Step 4: 提炼公告内容

基于 Step 2 的改动分析，生成公告摘要和更新点列表。

**摘要**（一句话，30 字以内）：概括本轮最重要的改动。
**更新点**（JSON 数组，每条一行）：每项改动一句话描述，面向用户（非技术术语）。
**未来规划**（JSON 数组）：从当前待办中提取 2-3 条，用 "TODO 待补充" 作为默认值。

**示例**：
```json
{
  "version": "v2.1.0",
  "summary": "新增发版自动化工具，修复师德墙发帖滚动问题",
  "updates": [
    "新增发版助手 skill，发版流程自动化",
    "修复师德墙发帖后新卡片只露出半张的问题"
  ],
  "plans": [
    "角色精灵表生成（R-003）",
    "TODO 待补充"
  ]
}
```

#### Step 5: 修改文件

**按顺序修改以下文件**：

1. **`package.json`**（根）：修改 `version` 字段（不带 v 前缀，如 `"3.0.0"`）
2. **`server/package.json`**：同上，保持一致
3. **`server/prisma/seedVersion.js`**：在 `versionList` 数组**最顶部**（第一个元素）插入新版本记录。`date` 用当天日期，`version` 带 v 前缀，`updates` 和 `plans` 用 `JSON.stringify()` 包装
4. **`CHANGELOG.md`**（根）：在第一条记录前插入新发版条目，格式化为标准架构级记录
5. **`.ai/handoff.md`**：更新 `最后更新` 时间、`稳定版本` commit（待提交后填写）、`当前阶段` 的版本号描述

#### Step 6: 确认并提交

修改完成后，展示所有改动文件的 diff 摘要，让用户确认：

```
=== 发版准备完成 ===
当前版本：v2.0.0 → 新版本：v2.1.0

改动文件：
  package.json       — version: 2.0.0 → 2.1.0
  server/package.json — version: 2.0.0 → 2.1.0
  seedVersion.js      — 新增 v2.1.0 版本记录
  CHANGELOG.md        — 新增发版条目
  handoff.md          — 更新版本号

确认后执行：git add -A && git commit -m "[deploy](发版): v2.1.0 ..." && git push
```

**必须等用户确认后**才能执行 git commit + push。遵循部署纪律：**不自动部署**，只提交代码。

## 边界约束

| 约束 | 要求 |
|------|------|
| **不猜测改动** | 无法从会话确认改动时，必须问用户 |
| **不自动部署** | 只提交代码，不执行 deploy.sh 或 scp |
| **不跳过确认** | 修改文件前先展示改动分析，修改后展示 diff 摘要 |
| **版本号规则** | 严格遵循 ADR-004：混合发版只递增最高级别 |
| **纯文档/重构时** | 提醒用户：本轮无功能变更，版本号不递增，只更新 CHANGELOG |
| **三者一致** | package.json(不带v) = 数据库Version(带v) = 线上版本 |
| **seedVersion 格式** | 新版本记录插入数组**最顶部**（倒序，最新在上） |
| **CHANGELOG 格式** | 遵循根 CHANGELOG.md 现有格式（架构级记录，含决策依据/替代方案/影响评估） |
